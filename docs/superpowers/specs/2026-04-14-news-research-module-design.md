- **文档版本**：v1.0
- **日期**：2026-04-14
- **作者**：Claude + 朱煜
- **目标用户**：西南政法大学新闻传播学院老师（及其研究生团队）
- **截止节点**：2026 年 4-5 月两份研究报告所需数据
- **关联分支**：`feature/news-research-module`（待开）

## 1. 概述与目标

### 1.1 一句话定义

> 一个面向新闻传播学研究者的**多维度新闻检索 · 命中统计 · 研究报告生成**工作台：以人工登记的四级媒体源为样本池，围绕主题关键词做全网和白名单站点采集，提供知网式高级检索、分级聚合统计、Excel/Word 研究报告导出。

### 1.2 设计目标

| 目标 | 衡量标准 |
|---|---|
| **数据可信** | 每条报道可点击原链核查；媒体层级与区县归属在采集源头就打好标签 |
| **可复现** | 同一检索条件、同一时间窗，两次查询结果一致（关键词命中 + 固定向量模型） |
| **聚合口径清晰** | 支持「按关键词去重」「按区县去重」两级去重，与老师现有报告口径完全对齐 |
| **可演进** | 媒体源库、主题词库、区县清单均为数据表驱动，老师可在后台自助维护 |
| **4-5 月交付** | 第一期 MVP 必须能产出 2025 全年 × 39 区县 × 16 主题 × 四级媒体的数据与初稿报告 |

### 1.3 非目标（YAGNI）

第一期**不做**以下：

- 微博 / 微信公众号 / 抖音等**社交平台**采集（法务 + 反爬，风险太高，老师也没明确要）
- **线下宣传活动**数据（老师报告里的「六五环境日活动 127 场」这类由老师手填或另开表单，不抓）
- **舆情情感分析**（正负向）
- **跨语种**检索
- 研究任务之间的**共享协作**（先做单用户私有，权限与 RBAC 对齐即可）

## 2. 用户场景与使用流程

### 2.1 典型场景 A：跑季度/年度报告

```
老师新建研究任务「2025 年全年重庆市生态文明传播报道分析」
  → 选时间段 2025-01-01 ~ 2025-12-31
  → 勾选 16 个主题（生态文明传播主题预置组）
  → 勾选 39 个区县（全选）
  → 勾选四级媒体（全选）
  → 可选：粘贴一批重要 URL 手动补数据
  → 提交任务
系统后台（Inngest）批量跑采集 + 命中分析
完成后 → 老师收到通知
  → 查看结果页：命中数据表 + 三张核心图表 + 两级去重统计
  → 导出 Excel（含透视表）+ Word 初稿报告
```

### 2.2 典型场景 B：临时专题检索

```
老师在检索页按知网式条件构造器组合：
  主题 = 「绿色发展」 OR 「低碳经济」
  AND 区县 = 「九龙坡区」 OR 「两江新区」
  AND 媒体层级 = 「央级」
  AND 时间 = 2025-06~2025-12
→ 得到命中列表 → 看统计 → 导出 Excel
```

### 2.3 典型场景 C：维护媒体源库

```
老师/管理员进入「媒体源管理」后台：
  → 新增媒体「万州时报」，层级 = 区县级，所属区县 = 万州区
  → 填官网 https://wzsb.cqwz.gov.cn
  → 填列表页 URL 模板（系统按模板抓列表）
  → 保存，系统启用该站点常态采集
```

## 4. 数据模型

### 4.1 表结构总览

新增 **11 张表**，均在 `src/db/schema/research/` 下，与现有表隔离但可引用：

```

research_topic_keywords          主题关键词（共词 + 近似称谓）

research_task_targets            任务的采集目标（主题×区县×媒体层级的组合网格）
```

## 5. 采集架构

### 5.1 三路采集

```
[采集源]              [技术栈]                    [打标时机]
──────────────────────────────────────────────────────────
全网 Tavily 搜索    → Tavily API → Jina Reader  → 抓回后按域名匹配 media_outlet_aliases
白名单站点常态采集  → Inngest cron + Jina       → 直接绑定 outlet_id
URL 临时粘贴       → 队列逐个 Jina Reader       → 按域名匹配；无匹配标为「未分类」，
                                                  由老师后台补登记
```

**全部三路的产出统一写入 `news_articles`**，`source_channel` 字段区分来源。

### 5.2 Inngest 作业

新增或改造 4 个作业（放 `src/inngest/functions/research/`）：

| 作业 | 触发 | 职责 |
|---|---|---|
| `research-task-start` | 事件 `research/task.submitted` | 编排：生成采集目标网格 → fan-out 到下列三个作业 |
| `research-tavily-crawl` | 事件 `research/tavily.crawl` | 按「主题关键词 × 时间段 × 站点过滤」调 Tavily，抓回 URL，拉 Jina Reader，入库 `news_articles` |
| `research-whitelist-crawl` | 定时 + 事件 | 按 `media_outlet_crawl_configs` 抓白名单站点 |
| `research-article-analyze` | 事件 `research/article.enqueue_analyze` | 向量化 + 关键词命中扫描 + 语义命中计算，写 `news_article_topic_hits` |
| `research-task-finalize` | 事件 `research/task.all_done` | 聚合统计、生成 `result_summary`、产出 Excel+Word 文件、通知用户 |

**并发与配额**：
- Tavily 每分钟配额有限：作业层加令牌桶限流
- Jina Reader 大批量：复用现有重试退避
- 一个研究任务预计采集 **5000-20000 篇** 文章，分析阶段单篇 ~300ms（嵌入+关键词），全流程预计 **30 分钟 - 2 小时**

### 5.3 已有通道复用

- **Tavily 搜索**：`src/lib/agent/tools/web-search.ts` 已封装 → 研究模块增加薄包装支持「站点白名单过滤」
- **Jina Reader**：`src/lib/web-fetch.ts:181` 已有 → 直接复用
- **Jina Embeddings**：`src/lib/knowledge/embeddings.ts` 已有 → 复用，切换模型参数
- **Inngest 基础设施**：已就绪，只加新函数

## 6. 关键词命中 + 语义命中

### 6.1 关键词命中算法

```
对每篇文章 article，对每个研究任务选中的 topic：
  对 topic 的所有 keywords（含共词 + 近似称谓）：
    在 article.title 做正则命中（边界匹配，避免「低碳」命中「降低碳排」，中文需定制切分）
    在 article.content 做同样命中
  如果 ≥1 keyword 命中：
    写入 hit 记录，match_type=keyword
    matched_keywords = 实际命中到的关键词列表
    matched_fields = ['title', 'content'] 中命中的
```

**中文分词**：第一期用**词典+正则**（不引入 jieba 等依赖，避免运维负担）。规则：
- 关键词整串匹配（不分词）
- 关键词前后字符不能是中英文连续字符（`[\u4e00-\u9fff a-zA-Z0-9]`）→ 模拟「词边界」

### 6.2 语义命中算法

```
加载 topic 的 samples，取各 sample 的 embedding 求均值作为 topic_centroid
对每篇 article，计算 cosine(article.embedding, topic_centroid)
如果 >= semantic_threshold（默认 0.72）：
  写入 hit 记录，match_type=semantic（或 both 如果同时也命中关键词）
  semantic_score = 实际相似度值
```

**阈值**：默认 0.72，老师可在任务创建时调整（0.60-0.85 区间滑条）。

**为什么向量相似度能保证可复现**：
- 用固定版本 `jina-embeddings-v3`
- 样本向量和文章向量都持久化，不在线实时算
- 阈值作为任务参数固化
→ 老师论文里可以明确写「使用 Jina v3 模型，主题语义命中阈值 0.72」

### 6.3 两种命中的组合

- `match_type = 'keyword'`：仅关键词命中
- `match_type = 'semantic'`：仅语义命中（老师可看到「语义召回但关键词未命中」的文章，作为论文讨论素材）
- `match_type = 'both'`：同时命中（最可信）

## 7. 去重与聚合统计

### 7.1 两级去重口径

对应老师现有报告的描述：

```
Step 1（原始）：
  SUM(hits)                             → 例：82975 条（含主题间重复）

Step 2（按关键词去重）：
  每篇文章对每个主题只算 1 次，
  但一篇文章可以同时被计入多个主题：
  SELECT COUNT(DISTINCT (article_id, topic_id))
  → 例：29154 条

Step 3（按区县去重，原文是「去除关键词重复后」）：
  每篇文章只算一次（不管它命中几个主题）
  SELECT COUNT(DISTINCT article_id)
  → 例：9759 条

Step 4（跨区县去重）：
  一篇文章若被多个区县的媒体同时提到（转载），只算一次：
  SELECT COUNT(DISTINCT url_hash)
  → 例：6032 条
```

系统在任务完成后**一次计算好这四个数**存入 `research_tasks.result_summary`，报告直接引用。

### 7.2 核心聚合指标

| 指标 | SQL 口径 |
|---|---|
| 各主题报道量 | `GROUP BY topic_id COUNT(DISTINCT article_id)` |
| 各区县报道量 | `GROUP BY district_id_snapshot COUNT(DISTINCT article_id)` |
| 各媒体层级报道量 | `GROUP BY outlet_tier_snapshot COUNT(DISTINCT article_id)` |
| 主题 × 区县 交叉 | `GROUP BY topic_id, district_id_snapshot` |
| 主题 × 媒体层级 交叉 | `GROUP BY topic_id, outlet_tier_snapshot` |
| Top/Bottom 3 主题 | 上述主题排序 DESC/ASC LIMIT 3 |
| Top/Bottom 3 区县 | 同上 |
| 单家媒体发文量 | `GROUP BY outlet_id` |

## 12. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| **区县融媒体抓不到** | 数据覆盖不全 | 第一期只保证「网站形态可抓」的区县媒体；抓不到的（仅公众号/客户端）标注「未覆盖」，老师报告中明示 |
| **Tavily 返回的域名匹配不到已登记媒体** | 大量「未分类」数据堆积 | 提供「未分类报道列表」后台，老师可以人工把域名补进 `media_outlet_aliases` |
| **向量化成本** | Jina API 费用 | 先估算全年 39 区县的日均发文量（预计 2000-5000 条），月均嵌入成本可控；设计任务级限流 |
| **中文关键词命中的误命中** | 「低碳」命中「降低碳排」、「美丽中国」命中「中国很美丽」等 | 采用词边界正则 + 人工复核界面；老师可在词库管理处为关键词加「排除词」白名单（v1.1 增强） |
| **研究任务跑几小时中断** | 重跑代价大 | Inngest 本身支持 step 级幂等，每批采集 + 分析作为 step，失败重试不重复入库 |
| **老师报告口径与系统统计偏差** | 交付质量受损 | 第一期请老师**用一份已有报告做对照验证**，跑 2024 年 1-10 月数据，对比老师原报告中的数字，误差率要求 < 5% |
