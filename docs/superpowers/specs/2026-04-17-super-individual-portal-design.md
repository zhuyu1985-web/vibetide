## 2. 首页门户设计

### 2.1 设计理念

首页是"超级个体的智媒工作空间"——输入框为绝对主体，用户一进来就能直接下达指令。

### 2.2 布局结构（自上而下）

```
┌─────────────────────────────────────────────────┐
│  状态标签：[●] 超级个体已就绪 · 8位专家待命        │
│                                                   │
│  标题：你的智媒工作空间                             │
│  副标题：与 AI 团队协作，高效完成内容生产            │
│                                                   │
│  ┌─────────────────────────────────────────────┐ │
│  │  统一指令输入框（主体）                       │ │
│  │  placeholder: "告诉超级个体你想做什么..."      │ │
│  │  [🎙语音] [📎附件] [DeepSeek▾]      [↑发送]  │ │
│  └─────────────────────────────────────────────┘ │
│                                                   │
│  AI 专家团队（横向滚动）          [全部员工 →]      │
│  [📡热点分析师] [💡选题策划师] [📦素材研究员] ...   │
│                                                   │
│  场景快捷启动（3×2 网格）        [+ 自定义场景]     │
│  [🏛两会报道团] [🏃马拉松直击队] [⚡突发应急组]      │
│  [🎯主题宣传队] [🏘民生服务组]   [📰快发流水线]      │
│                                                   │
│  动态（2列）                                       │
│  [🔄 进行中的任务]    [💬 最近对话]                 │
└─────────────────────────────────────────────────┘
```

### 2.3 核心动画效果

| 元素 | 动画 | 实现方式 |
|------|------|---------|
| 状态标签绿点 | 呼吸脉冲 | CSS `@keyframes pulse` |
| 输入框 | hover 发光边框 + 阴影扩散 | CSS transition |
| 员工头像 | 专属微动效（见3.3节）+ hover上浮 | Lottie / CSS animation |
| 场景卡片 | hover 发光边框 + translateY(-2px) | CSS transition |
| 背景 | 粒子系统 + 渐变光晕 | Canvas（从 `src/app/landing/components/hero-background.tsx` 提取为 `src/components/shared/ParticleBackground.tsx` 共享组件） |
| 模式切换 | 用户发送消息后 Hero 区收起，无缝过渡到对话界面 | Framer Motion layoutAnimation |

### 2.4 输入框智能路由

用户输入内容后，利用现有 `intent-recognition.ts`：
- 自动识别意图，推荐匹配的场景或员工
- 高置信度（>0.8）：直接开始执行
- 中置信度（0.5-0.8）：展示确认卡片
- 低置信度（<0.5）：询问用户意图

### 2.5 场景卡片交互

点击场景卡片 → 展开 Sheet/SlideOver：
- 场景说明 + 参与员工头像
- 工作流步骤可视化（Pipeline）
- 输入表单（场景专属字段）
- 底部双按钮：
  - **[一键启动]** → 创建 Mission，自动执行全部步骤
  - **[进入对话]** → 跳转对话中心，团队已组好，手动逐步指挥

### 3.2 头像方案

- **风格**：3D 卡通拟人化（类似 Apple Memoji / Notion 风格）
- **素材**：AI 生成（Midjourney/DALL-E）3D 卡通人物，每人配有职业特征（数据眼镜、话筒、笔等）
- **格式**：静态 PNG（用于小尺寸场景）+ Lottie JSON（用于首页/详情页动效）
- **存放**：`public/avatars/{slug}.png` + `public/avatars/{slug}.json`
- **过渡方案**：Lottie 资源未就绪前，使用 PNG 静态头像 + CSS 微动效（见3.3节）作为 fallback。组件内通过 `useMemo` 检测 `.json` 文件是否存在，自动降级

## 4. 6大预设场景工作流

### 4.1 场景总览

| # | 场景名称 | 团队配置 | 核心特色 | 时效目标 |
|---|---------|---------|---------|---------|
| 1 | 两会报道团 | 热点分析师 + 选题策划师 + 内容创作师 + 质量审核官 | 四级内容流水线、政治敏感度检查 | 快讯10min |
| 2 | 马拉松直击队 | 热点分析师 + 内容创作师 + 视频制片人 | 即拍即发、并行执行、二次传播 | 关键镜头5min |
| 3 | 突发应急组 | 热点分析师 + 内容创作师 + 质量审核官 | 信源可信度标注、谣言识别、紧急模式审核 | 首发15min |
| 4 | 主题宣传队 | 选题策划师 + 内容创作师 + 视频制片人 + 渠道运营师 | 模块化拆解、IP化、多平台适配 | 按日历排期 |
| 5 | 民生服务组 | 热点分析师 + 内容创作师 | 2h闭环、竖屏适配、反馈追踪 | 线索到发布2h |
| 6 | 快发流水线 | 热点分析师 + 内容创作师 + 渠道运营师 | 标准化模板、三波次推送、先网后台 | 首发率90% |

### 4.2 场景一：两会/重大时政会议报道团

**触发输入**：会议名称、关注议题（可选）、输出渠道偏好

**工作流**：

```
Step 1: 热点分析师 → 全网扫描会议相关热点、政策要点、舆情焦点
   输出: 热点清单 + 关键政策摘要

Step 2: 选题策划师 → 基于热点设计报道角度
   输出: 选题方案（含四级内容流水线分配：快讯/短视频/深度/数据新闻）

Step 3: 内容创作师 → 按选题方案分级生产
   ├─ 快讯稿（10min出稿）
   ├─ 短视频脚本（30min）
   ├─ 深度解读稿（2h）
   └─ 数据新闻图文
   输出: 多级内容包

Step 4: 质量审核官 → 政治合规审核 + 事实核查 + 三审留痕
   输出: 审核报告 + 修订建议

Step 5: 内容创作师 → 根据审核意见修订
   输出: 终稿内容包
```

**特色能力**：政治敏感度检查（自动标记涉政表述）、四级内容流水线

### 4.3 场景二：马拉松/体育赛事直击队

**触发输入**：赛事名称、赛事时间、重点关注（冠军冲线/人物故事/城市风貌）

**工作流**：

```
Step 1: 热点分析师 → 赛事信息汇总（参赛阵容、赛道路线、历史数据）
   输出: 赛事背景资料包

Step 2: 内容创作师 → 预热内容生产
   ├─ 赛前预告稿
   ├─ 选手/城市故事稿
   输出: 预热内容

Step 3: [赛事进行中] 并行执行：
   ├─ 内容创作师 → 即时快讯（关键节点5min出稿）
   ├─ 视频制片人 → 高光片段脚本 + 剪辑方案
   └─ 热点分析师 → 实时舆情监控 + 话题热度
   输出: 快讯流 + 视频脚本 + 热度报告

Step 4: 内容创作师 → 赛后总结
   ├─ 纪录片脚本
   ├─ 人物专访稿
   └─ 城市宣传文案
   输出: 二次传播内容包
```

**特色能力**："即拍即发"模式、并行执行、长尾内容二次传播

### 4.4 场景三：突发事件/灾害应急组

**触发输入**：事件描述、事件类型（自然灾害/安全事故/公共卫生）、紧急程度

**工作流**：

```
Step 1: 热点分析师 → 紧急信息采集
   ├─ 事件经过、官方通报、现场信息、社交媒体
   输出: 事件信息汇总（标注信源可信度）

Step 2: 质量审核官 → 信息核实
   ├─ 交叉验证信源
   ├─ 识别谣言
   输出: 核实报告（标注已确认/待确认/疑似谣言）

Step 3: 内容创作师 → 分级生产（简化审核，特事特办）
   ├─ 快讯首发（15min内）
   ├─ 滚动更新
   ├─ 辟谣/预警信息
   └─ 服务型内容（避险手册/求助通道）
   输出: 应急内容流

Step 4: 热点分析师 → 持续追踪
   ├─ 救援进展、舆情变化、后续处理
   输出: 追踪更新

Step 5: 循环 Step 2→4 直到事件平息
```

**特色能力**：信源可信度标注、谣言识别、简化审核流程（紧急模式）、服务型内容自动生成、持续追踪循环

### 4.5 场景四：主题宣传/重大节庆宣传队

**触发输入**：宣传主题、目标受众、传播周期、重点渠道

**工作流**：

```
Step 1: 选题策划师 → 主题拆解 + 创意策划
   ├─ 拆解为多个短视频单元（模块化生产）
   ├─ 设计系列化IP方案
   └─ 制定传播节奏表
   输出: 策划方案 + 内容日历

Step 2: 内容创作师 → 批量内容生产
   ├─ 短视频脚本（15-60秒，碎片化传播）
   ├─ 长视频/纪录片脚本
   ├─ 图文/海报文案
   输出: 内容素材包

Step 3: 视频制片人 → 视觉方案设计
   ├─ 视频分镜/剪辑方案
   ├─ 封面/海报设计方案
   输出: 视觉制作方案

Step 4: 渠道运营师 → 分发策略
   ├─ 多平台适配方案（抖音/视频号/微博/微信）
   ├─ KOL联动方案
   ├─ 话题运营策略
   输出: 分发执行方案
```

**特色能力**：模块化拆解 + 批量生产、系列化IP打造、多平台适配、内容日历排期

### 4.6 场景五：民生新闻/城市服务组

**触发输入**：民生线索/话题、涉及领域（交通/医疗/教育/消费/环境）、报道类型

**工作流**：

```
Step 1: 热点分析师 → 线索深挖
   ├─ 相关政策梳理
   ├─ 类似案例检索
   ├─ 舆情热度分析
   输出: 选题调研报告

Step 2: 内容创作师 → 内容生产
   ├─ 短视频稿（1-3分钟，竖屏、口语化、字幕突出）
   ├─ 图文稿（500-1000字，实用优先）
   └─ 服务型内容（政策解读/办事指南/生活贴士）
   输出: 民生内容包

Step 3: 热点分析师 → 反馈追踪
   ├─ 评论区舆情分析
   ├─ 问题解决进展追踪
   输出: 追踪报告（形成 报道→反馈→解决 闭环）
```

**特色能力**：2小时线索到发布闭环、竖屏适配、服务型内容自动生成、反馈闭环追踪

### 4.7 场景六：日常新闻/新媒体快发流水线

**触发输入**：新闻线索/通稿、发稿优先级（紧急/重点/一般）、目标渠道

**工作流**：

```
Step 1: 热点分析师 → 信息汇聚 + 选题筛选
   ├─ 通稿/线索/网络信息聚合
   ├─ 按优先级分类（紧急/重点/一般）
   输出: 选题清单（含优先级标签）

Step 2: 内容创作师 → 标准化快速生产
   ├─ 新媒体首发稿（标题/正文/配图建议）
   ├─ 多平台适配版本（微博140字/微信图文/短视频口播稿）
   输出: 多版本内容包

Step 3: 渠道运营师 → 分发排期
   ├─ "三波次推送"策略（求快→求全→求深）
   ├─ 最佳发布时间建议
   ├─ 各平台格式适配检查
   输出: 分发执行计划
```

**特色能力**：标准化流水线（固定格式模板）、三波次推送策略、先网后台机制、90%新媒体首发率目标

### 4.8 场景技术实现

每个场景对应一个 `SCENARIO_CONFIG` 条目 + 一个 `WorkflowTemplate` 记录：
- `SCENARIO_CONFIG` 定义场景元数据（名称、图标、颜色、默认团队、描述）
- `WorkflowTemplate` 定义步骤链（每个 step 指定 employeeSlug、skills、inputSchema、dependsOn）
- 场景启动时，系统基于模板创建 `Mission` + `MissionTask` 实例

场景的 `employeeScenarios` 表记录保持不变，用于对话中心内的员工专属场景。首页场景卡片直接映射到 `SCENARIO_CONFIG` 中的6个预设。

**与现有场景的关系**：现有 `SCENARIO_CONFIG` 中的8个泛化场景（`breaking_news`, `flash_report`, `press_conference` 等）保留不动，仍可在对话中心和任务创建中使用。首页6个新场景是面向具体媒体生产的"高级场景"，使用新的 key（`lianghui_coverage`, `marathon_live`, `emergency_response`, `theme_promotion`, `livelihood_service`, `quick_publish`）。两套场景共存，不做迁移。

**突发应急组的"循环追踪"实现**：现有 Mission 系统是 DAG 结构，不支持循环。突发场景的"循环 Step 2→4"通过以下方式实现：每轮追踪完成后，系统创建一个新的 follow-up Mission（标记 `parentMissionId` 关联），而非在同一 Mission 内循环。用户也可以在对话模式中手动触发下一轮追踪。

## 5. 自定义员工 & 场景

### 5.1 自定义员工（GPTs 式轻量创建）

**入口**：首页员工列表 "+" 按钮 / 员工管理页"创建员工"

**创建流程**（3步向导）：

```
Step 1: 基础信息
  ├─ 选择基础角色模板（8个预设中选一个作为基底）
  ├─ 自定义名称
  ├─ 选择/上传头像
  └─ 一句话描述

Step 2: 能力配置
  ├─ 指令设定（textarea，预填模板提示）
  ├─ 技能继承（基底角色技能默认勾选，可增减）
  └─ 绑定知识库（可选）

Step 3: 预览 & 发布
  ├─ 试对话（快速验证效果）
  └─ 发布到团队 / 仅自己可见
```

**技术实现**：
- 存入 `ai_employees` 表，使用现有 `is_preset` 字段（`is_preset = 0` 表示自定义员工，无需新增字段）
- 自定义员工的 slug 使用 `custom_{uuid}` 格式。前端需将 `EmployeeId` 类型从封闭 union 放宽为 `string`，`EMPLOYEE_META` 查找逻辑增加 DB fallback（预设员工走内存常量，自定义员工走数据库查询）
- 自定义 instructions 注入 agent assembly pipeline 第一层 Identity prompt
- 技能绑定复用 `employee_skills` 表
- 知识库绑定复用 `employee_knowledge_bases` 表

### 5.2 自定义场景

**路径A：基于预设场景调参**

选择预设场景 → 修改团队成员 → 调整步骤 → 修改输入表单 → 保存为"我的场景"

**路径B：工作流编排器创建**

进入工作流中心 → 新建工作流 → 可视化编排（复用现有 workflow builder）→ 保存后出现在首页场景区域

**入口统一**：首页"+ 自定义场景"按钮，弹出选择：
- "基于现有场景修改" → 路径A
- "从零创建工作流" → 跳转工作流中心

## 6. 钉钉/微信渠道打通

### 6.1 架构

```
┌─────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  钉钉机器人   │────→│  Channel Gateway  │←────│  企业微信机器人   │
│  (Webhook)   │←────│  (API Route)     │────→│  (Webhook)      │
└─────────────┘     └────────┬─────────┘     └─────────────────┘
                             │
                    消息标准化层
                    ├─ 入站：外部消息 → 统一 Message 格式
                    ├─ 路由：意图识别 → 分发到对应员工/场景
                    └─ 出站：Mission结果 → 适配各渠道格式
                             │
                    ┌────────▼─────────┐
                    │  现有核心引擎      │
                    │  Intent Recognition + Agent Execution + Mission System
                    └──────────────────┘
```

### 6.2 功能范围

**入站**：
- 钉钉/微信群 @机器人 → 意图识别 → 自动分配员工处理
- 文字指令 + 图片/文件附件
- 快捷指令（如 `#两会报道 主题:政府工作报告`）

**出站**：
- Mission 状态变更通知
- 任务产出物推送（适配渠道格式和字数限制）
- 审核节点提醒

**消息格式适配**：

| 内容类型 | 钉钉 | 企业微信 |
|---------|------|---------|
| 文本 | Markdown 卡片 | 文本/Markdown |
| 长文 | 截断 + "查看全文"链接 | 截断 + 链接 |
| 文件 | 文件卡片 | 文件消息 |
| 任务进度 | ActionCard（带按钮） | 模板卡片消息 |

### 6.3 安全与容错

**Webhook 签名验证**：
- 钉钉：HMAC-SHA256 签名验证（使用 `DINGTALK_ROBOT_SECRET` + timestamp）
- 企业微信：AES 消息加解密（使用 `WECHAT_WORK_ENCODING_AES_KEY`）+ URL Token 验证

**多租户路由**：每个 `channel_configs` 记录绑定一个 `organization_id`。Webhook URL 中包含 channel config ID（如 `/api/channels/dingtalk/webhook/{configId}`），网关据此确定所属组织。

**容错处理**：
- 意图识别失败 → 回复"我没有理解您的指令，请尝试更具体的描述"+ 场景快捷指令提示
- 消息发送限流 → 使用 Inngest 队列化出站消息，自动退避重试
- 平台 API 异常 → 记录失败日志，不阻塞 Mission 流程

### 6.4 技术实现

新增路由：
- `src/app/api/channels/dingtalk/webhook/[configId]` — 钉钉消息回调
- `src/app/api/channels/wechat/webhook/[configId]` — 企业微信消息回调
- `src/app/api/channels/notify` — 统一出站推送

新增 DB 表：
- `channel_configs` — 渠道配置（organization_id, platform, webhook URL、密钥、启用状态）
- `channel_messages` — 消息记录（入站/出站、关联 mission_id、organization_id）

## 7. 媒体合规体系

### 7.1 三审三校架构

```
内容产出
  → 初审（AI 自动）
     ├─ 质量审核官执行：政治合规、敏感词、事实核查
     ├─ 输出：审核报告（问题列表 + 严重等级 + 修改建议）
     └─ 自动标记：🟢通过 / 🟡需修改 / 🔴需人工

  → 复审（可配置：AI 或人工）
     ├─ AI模式：换一组审核规则重新检查（交叉验证）
     ├─ 人工模式：推送到指定审核人，系统内审批
     └─ 记录：审核人、时间、意见、修改对比

  → 终审（可配置：AI 或人工）
     ├─ 最终确认，签发放行
     └─ 记录：终审人、签发时间、最终版本快照
```

**可配置性**：
- 不同场景设置不同审核严格度（突发应急 = 快审模式，仅政治合规 + 事实准确）
- 组织级别可自定义审核规则和敏感词库

### 7.2 审核规则引擎

6个审核维度，每个独立评分：

| 维度 | 说明 |
|------|------|
| political_compliance | 政治合规（涉政表述、领导人姓名/职务准确性） |
| factual_accuracy | 事实准确（数据、引用、时间、地点） |
| sensitive_content | 敏感内容（涉民族/宗教/暴力/隐私） |
| language_quality | 语言质量（错别字、语病、标点） |
| source_credibility | 信源可信度（标注来源、交叉验证） |
| legal_compliance | 法律合规（著作权、肖像权、隐私权） |

### 7.3 全流程留痕

覆盖内容生命周期每个节点：

```
选题 → 采编 → 初审 → 复审 → 终审 → 发布 → 追踪
 [log]  [log]  [log]  [log]  [log]  [log]  [log]
```

每条留痕记录：

| 字段 | 说明 |
|------|------|
| operator | 操作者（AI员工slug 或 人工用户ID） |
| operator_type | `ai` / `human` |
| action | create / edit / review / approve / reject / publish |
| stage | planning / writing / review_1 / review_2 / review_3 / publishing |
| timestamp | 精确到秒 |
| content_snapshot | 操作时的内容快照 |
| diff | 与上一版本的差异（JSON diff） |
| comment | 操作说明/审核意见 |
| metadata | 扩展信息（审核分数、问题列表等） |

### 7.4 审核中心

新增 `/audit-center` 路由：

```
审核中心
├─ 待审列表（按紧急度排序，支持筛选场景/员工/阶段）
├─ 审核详情页
│   ├─ 左栏：内容预览（支持 diff 对比模式）
│   ├─ 右栏：AI 审核报告（问题列表、严重等级、修改建议）
│   └─ 底部：审核操作栏（通过/退回/修改意见）
├─ 审核历史（全流程时间线视图，可展开每个节点查看详情）
└─ 审核统计（通过率、平均审核时长、常见问题类型分布）
```

### 7.5 审核与 Mission 系统集成

**触发时机**：当 Mission 中"质量审核官"的 task 完成时，自动创建 `audit_records` 记录（初审）。如果审核配置为人工复审/终审，将对应 MissionTask 状态设置为新增的 `awaiting_review`（需在 `missionTaskStatusEnum` 中添加），Mission 管线在此暂停。

**审核流转**：
```
MissionTask(质量审核官) 完成
  → 创建 audit_record (stage=review_1, result=auto)
  → 检查场景审核配置：
     ├─ 全自动：自动创建 review_2 audit_record → review_3 → 继续 Mission
     ├─ 需人工：设置下一个 task 为 awaiting_review → 推送到审核中心
     └─ 快审模式（突发场景）：仅 review_1，跳过 review_2/3
```

**退回处理**：审核中心"退回"操作 → 在当前 Mission 中创建一个新的修订 task（assignee=内容创作师，inputContext=审核意见），而非重开已完成 task。修订完成后重新进入审核流程。

### 7.6 审核评分规则

每个维度使用三级评分：`pass`（通过）/ `warning`（需关注）/ `fail`（不通过）。综合判定逻辑：
- 任一维度 `fail` → 整体标记 🔴需人工
- 任一维度 `warning` 且无 `fail` → 整体标记 🟡需修改
- 全部 `pass` → 整体标记 🟢通过

### 7.7 数据模型

新增表：
- `audit_records` — 审核记录（关联 mission_id/article_id、阶段、结果、意见）
- `audit_rules` — 审核规则配置（组织级、场景级）
- `content_trail_logs` — 全流程留痕日志
- `sensitive_word_lists` — 自定义敏感词库
