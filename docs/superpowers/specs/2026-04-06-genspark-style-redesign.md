# Vibetide 产品改进设计：借鉴 Genspark.ai 交互范式

> 日期：2026-04-06
> 状态：草案
> 参考：Genspark AI Workspace 3.0

## 1. 背景与目标

### 1.1 现状痛点

- 登录后直接进入任务列表页，缺乏统一的首页入口和全局交互点
- 侧边栏 6 个分组、20+ 菜单项，导航层级深、认知负担重
- AI 员工展示为简单的状态卡片列表，用户难以直观理解每个员工能做什么
- 工作流引擎（Inngest/Mission）仅后台运行，用户无法自定义创建和编辑工作流
- 对话中心是唯一的 AI 交互入口，缺乏快捷触达方式

### 1.2 改进目标

借鉴 Genspark.ai 的产品交互范式，同时**保持媒体行业垂直定位**：

1. **统一首页入口**：自然语言输入框 + AI 员工快捷面板，降低使用门槛
2. **精简导航**：8 个一级入口 + "更多" 收纳，降低认知成本
3. **智能体市场化展示**：让用户快速理解每个 AI 员工的能力并直接派发任务
4. **可视化工作流编辑**：支持内置场景模板 + 用户自定义工作流创建
5. **首页内嵌对话**：轻量任务在首页完成，深度交互可展开到独立页面

### 1.3 核心定位差异

| 维度 | Genspark | Vibetide（改造后） |
|------|----------|-------------------|
| 行业定位 | 通用 AI 工作空间 | 媒体行业 AI 内容生产平台 |
| 智能体 | 通用型（幻灯片/表格/文档等） | 媒体专属 8 个数字员工 + 自定义扩展 |
| 工作流模板 | 邮件/数据/营销等通用场景 | 突发新闻/发布会追踪/热点早报等媒体场景 |
| 输出物 | Sparkpages、PPT、表格 | 稿件、视频脚本、数据报告、多渠道发布 |

**交互流程：**

1. 用户在输入框输入自然语言需求
2. 意图识别系统（已有 `intent-recognition.ts`）分析输入，路由到对应员工
3. 首页展开内嵌对话面板，显示：匹配的员工、思考过程、技能调用、结果输出
4. 轻量任务在面板内完成；复杂任务显示"展开到独立页面"按钮，跳转到 `/chat?employee=xxx`
5. 需要多步骤协作的任务自动创建 Mission，用户可在任务中心查看进度

**员工图标卡片行：**

- 点击员工图标：直接进入与该员工的对话（输入框预填员工上下文）
- 最后一个是"全部员工"入口，跳转到 `/ai-employees`

### 2.4 工作流系统：场景模板 + 线性编辑器

#### 2.4.1 概念模型

```
工作流（Workflow）= 可复用的场景模板（流程定义）
  ├── 触发条件（定时 / 手动 / 事件）
  ├── 步骤序列（Step 1 → Step 2 → ... → Step N）
  │    └── 每步 = 员工 + 技能 + 输入参数
  └── 输出动作（保存 / 发布 / 通知）

Mission = 工作流的一次具体执行实例
  ├── 关联工作流模板（可选，也可由 AI 自动规划）
  ├── 执行上下文（输入数据、参数）
  └── 执行结果（产出物、状态）
```

**与现有系统的关系：**

- 工作流模板存储在新的 `workflow_templates` 表（已有该表，需扩展字段）
- 执行时创建 Mission 记录，复用现有 Mission 引擎（DAG 执行、Inngest 调度）
- 第一期 UI 只暴露线性编排，但底层数据结构按 DAG 设计（`steps` 中每步有 `dependsOn` 字段）

**编辑器功能：**

- **左侧面板**：对话式输入，描述想自动化的任务，AI 自动生成工作流步骤
- **中间面板**：线性步骤可视化，每步显示：序号、步骤名称、负责员工/工具、操作菜单（编辑/删除/上移/下移）
- **右侧面板**：添加步骤，按分类展示可选的员工、工具、输出动作
- **步骤配置**：点击步骤展开配置表单（选择员工、技能、输入参数、输出格式）
- **底部操作栏**：测试运行（干跑一次看效果）、添加步骤、开启/关闭自动执行、保存

**数据结构设计（DAG 兼容）：**

```typescript
interface WorkflowTemplate {
  id: string;
  name: string;
  description: string;
  category: 'news' | 'video' | 'analytics' | 'distribution' | 'custom';
  trigger: {
    type: 'manual' | 'scheduled' | 'event';
    config: ScheduleConfig | EventConfig | null;
  };
  steps: WorkflowStep[];
  isBuiltin: boolean;       // 内置模板 vs 用户自定义
  organizationId: string;
  createdBy: string;
}

interface WorkflowStep {
  id: string;
  order: number;            // 线性编辑器用的排序
  dependsOn: string[];      // DAG 预留：依赖的步骤 ID 列表
  type: 'employee' | 'tool' | 'output';
  config: {
    employeeId?: string;    // AI 员工 ID
    skillId?: string;       // 技能 ID
    toolId?: string;        // 内置工具 ID
    outputAction?: string;  // 输出动作类型
    parameters: Record<string, any>;  // 步骤参数
    inputMapping: Record<string, string>;  // 从上游步骤映射输入
  };
}
```

### 2.5 对话中心调整

**现状：** `/chat` 是主要 AI 交互入口，左侧员工列表 + 右侧对话面板。

**改造后定位：**

- 对话中心保留为深度交互和历史管理的场所
- 主要入口改为首页输入框 → 内嵌对话面板
- 对话中心增加"对话历史"管理功能（按员工分组、按时间排序、搜索）
- 从首页"展开到独立页面"后，跳转到类似 `/chat/[conversationId]` 的独立对话页
- 侧边栏不再单独列出"对话中心"入口，通过首页自然进入

## 3. 补充设计细节

### 3.1 RBAC 权限适配

Tab 聚合页面（创作中心、内容管理、数据分析）的权限处理：

- 现有 `MENU_PERMISSION_MAP` 按路由映射权限，合并后需要调整为**按 Tab 粒度**控制
- 每个 Tab 对应原路由的权限 key，如 `/creation?tab=inspiration` 仍使用原 `/inspiration-pool` 的权限
- 如果用户对某聚合页面下的所有 Tab 都无权限，则侧边栏隐藏该一级入口
- 如果用户只有部分 Tab 权限，进入页面后只显示有权限的 Tab
- 实现：扩展 `MENU_PERMISSION_MAP`，增加 `tabs` 子项映射

### 3.2 首页内嵌对话的会话持久化

**会话创建流程：**

1. 用户在首页输入框提交消息
2. 意图识别完成后，系统立即在 `saved_conversations` 表创建新记录（`status: 'active'`）
3. 返回 `conversationId`，前端用此 ID 管理消息流
4. 对话面板关闭/页面离开时，如果只有一轮对话且无有价值产出，标记为 `status: 'ephemeral'`（不在历史中显示）
5. 点击"展开到独立页面"时，跳转到 `/chat/[conversationId]`，会话无缝延续

**轻量 vs 复杂任务判定规则：**

- **轻量任务**（在首页面板内完成）：意图识别匹配到单个员工 + 单个技能，预计一轮对话可完成
- **复杂任务**（建议展开或创建 Mission）：匹配到多个员工/技能、需要多步骤协作、或用户显式要求创建任务
- 交给前端展示"展开"按钮，由用户自行决定是否展开；不强制跳转

### 3.3 Tab 页面的 URL 策略

聚合页面使用 **URL query params** 管理 Tab 状态：

- 格式：`/creation?tab=inspiration`、`/content?tab=articles`
- 默认 Tab：进入页面时显示用户有权限的第一个 Tab
- 浏览器前进/后退正确切换 Tab
- 侧边栏高亮：匹配一级路由即高亮（如 `/creation?tab=xxx` 都高亮"创作中心"）
- "更多"面板中的入口点击后跳转到对应聚合页面 + Tab 参数

### 3.4 对话历史的访问入口

侧边栏不单独列出"对话中心"，但对话历史可通过以下途径访问：

- **首页**："最近对话"快捷区域，显示最近 5 条对话，点击进入 `/chat/[conversationId]`
- **首页**："查看全部对话"链接，跳转到 `/chat`（对话历史列表页）
- **AI 员工市场页**：每个员工卡片上显示"历史对话"入口
- **全局搜索**：支持搜索对话内容

### 3.5 路由迁移与重定向

路由变更需要添加重定向（在 `next.config.ts` 或 middleware 中）：

| 旧路由 | 新路由 | 说明 |
|-------|-------|------|
| `/employee-marketplace` | `/ai-employees` | 员工市场页重命名 |
| `/team-hub` | `/home` | 首页替代 |
| `/inspiration-pool` | `/creation?tab=inspiration` | 合并到创作中心 |
| `/competitive-benchmarking` | `/creation?tab=benchmarking` | 合并到创作中心 |
| `/media-assets` | `/content?tab=assets` | 合并到内容管理 |
| `/articles` | `/content?tab=articles` | 合并到内容管理 |
| `/publishing` | `/analytics?tab=publishing` | 合并到数据分析 |

原路由保留 301 重定向，确保已有书签和分享链接不失效。

### 3.6 workflow_templates 表 Schema 迁移

现有 `workflow_templates.steps` 为简单的 `{key, label, employeeSlug, order}` jsonb 数组。需要扩展：

**新增列：**
- `category` — enum: `news`, `video`, `analytics`, `distribution`, `custom`
- `trigger` — jsonb: `{type, config}`
- `is_builtin` — boolean, 默认 false
- `created_by` — 引用 `user_profiles.id`
- `is_enabled` — boolean, 定时工作流的开关
- `last_run_at` — timestamp
- `run_count` — integer

**steps jsonb 结构升级：**
```
旧: [{key, label, employeeSlug, order}]
新: [{id, order, dependsOn, type, name, config: {employeeId, skillId, toolId, outputAction, parameters, inputMapping}}]
```

编写 idempotent 迁移脚本，保留旧数据兼容性（旧 steps 格式可通过迁移脚本转换为新格式）。

### 3.7 AI 自动规划工作流（第一期简化）

工作流编辑器左侧的"AI 自动规划"功能，第一期简化实现：

- 用户输入自然语言描述（如"每天早上自动生成热点早报"）
- 调用 LLM，提供可用员工列表 + 技能列表作为上下文
- LLM 返回结构化的步骤序列 JSON（员工 ID + 技能 ID + 步骤名称）
- 前端将 JSON 渲染为步骤序列，用户可手动调整后保存
- 第一期不做"测试运行"功能（标记为 coming soon），降低复杂度

### 3.8 内置模板的初始化方式

- 内置模板在 `src/db/seed.ts` 中定义，`is_builtin = true`
- 每次 `db:seed` 时 upsert（按 name + is_builtin 去重）
- 内置模板用户不可删除，但可以"复制为我的工作流"后自行修改
- 模板定义也在代码中维护（`src/lib/workflow-templates.ts`），方便版本管理
