```yaml
# .github/workflows/drift-check.yml 示例
name: Schema Drift Check
on: [pull_request, push]
jobs:
  drift:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '24' }
      - run: npm ci
      - run: |
          npx drizzle-kit generate --name ci_drift > /tmp/out.log 2>&1 || true
          if grep -q "Your SQL migration file" /tmp/out.log; then
            echo "❌ Schema drift 检测到未应用迁移"
            cat /tmp/out.log
            exit 1
          fi
        env:
          DATABASE_URL: ${{ secrets.DATABASE_URL }}
```

## 🧭 关键架构决策（你必须理解）

1. **场景 = 工作流**（用户强调过：AI 员工是多工作流能力的载体）
   - workflow_templates 表是**单一真相源**
   - /home "场景快捷启动" + /employee/[id] "日常工作流"是两个入口
   - /inspiration 热点"生成稿件"按钮是**一键启动 startTopicMission**，不选工作流

2. **双向同步**（`skills/*/SKILL.md` ⇌ `skills.content`；`workflows/*/SKILL.md` ⇌ `workflow_templates.content`）
   - 文件系统是团队 wiki 源
   - DB 是运行时 + UI 编辑源
   - sync 脚本保障一致性

3. **Migrations 纪律**：
   - 所有 schema 变更走 `drizzle-kit generate`，**不再手写 SQL**（避免再次技术债）
   - 遇到 drift：先 generate，再用 `make-migration-idempotent.py` 转幂等，再 `run-catchup-migration.ts` apply
   - 最后 `drizzle-kit generate` 必须输出 "No schema changes, nothing to migrate 😴"

4. **Git Workflow（强制单分支）**：
   - 只用 `main`，不开 feature/worktree
   - 每个 commit 独立可 build（tsc 全绿）
   - Linter/agent 偶尔自动创建 branch，要 merge 回 main 立刻清理
