## 1. 平台架构总览

### 1.1 路由分组

应用路由位于 `src/app/`，使用 Next.js Route Groups 划分为两个逻辑区域：

| 路由组 | 路径 | 说明 | 认证保护 |
|--------|------|------|----------|
| `(auth)` | `src/app/(auth)/` | 登录、注册、OAuth 回调 | 无 layout 保护 |
| `(dashboard)` | `src/app/(dashboard)/` | 21 个 Dashboard 页面 | `layout.tsx` 中执行认证检查，未登录重定向至 `/login` |

**根页面** `/` 的路由逻辑：
- 已认证用户 → 重定向至 `/team-hub`
- 未认证用户 → 重定向至 `/login`

### 1.2 Server/Client 组件模式

Dashboard 下所有页面遵循一致的服务端/客户端拆分模式：

```
page.tsx           — Server Component，调用 DAL 获取数据，作为 props 传递
*-client.tsx       — Client Component（"use client"），接收 props，处理全部交互逻辑
```

**示例：**

```
src/app/(dashboard)/team-hub/
├── page.tsx               ← Server Component: 调用 DAL 查询团队数据
└── team-hub-client.tsx    ← Client Component: 渲染 UI、处理用户交互
```

`page.tsx` 负责数据获取，`team-hub-client.tsx` 负责渲染与交互。两者通过 props 进行数据传递，保证服务端渲染与客户端交互的清晰边界。

### 1.3 数据流总览

```
┌─────────────────────────────────────────────────────────────┐
│                        读取路径                              │
│                                                             │
│  Server Page (page.tsx)                                     │
│       ↓                                                     │
│  DAL (src/lib/dal/)         ← 只读查询函数                   │
│       ↓                                                     │
│  Drizzle ORM                                                │
│       ↓                                                     │
│  Supabase PostgreSQL                                        │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                        写入路径                              │
│                                                             │
│  Client Component 触发表单提交 / 按钮操作                     │
│       ↓                                                     │
│  Server Actions (src/app/actions/)  ← "use server" 函数     │
│       ↓                                                     │
│  Drizzle ORM → Supabase PostgreSQL                          │
│       ↓                                                     │
│  revalidatePath() 刷新缓存                                   │
└─────────────────────────────────────────────────────────────┘
```

- **DAL**（`src/lib/dal/`）：只读查询函数，返回 UI 类型（`AIEmployee`、`Team` 等，定义在 `src/lib/types.ts`）。负责将 DB 行数据转换为前端接口格式。
- **Server Actions**（`src/app/actions/`）：负责写入操作，所有 Action 标记 `"use server"`，通过 `requireAuth()` 确保认证，完成后调用 `revalidatePath()` 使缓存失效。
- **Mock 数据**（`src/data/`）：19 个静态数据文件。尚未迁移至 DAL 的页面直接导入 mock 数据。

## 2. 认证与授权

### 2.1 Supabase 客户端分层

根据运行环境，使用三个不同的 Supabase 客户端实例：

| 客户端 | 文件路径 | 运行环境 |
|--------|----------|----------|
| Browser Client | `src/lib/supabase/client.ts` | 浏览器端 |
| Server Client | `src/lib/supabase/server.ts` | RSC / Server Actions |
| Middleware Client | `src/lib/supabase/middleware.ts` | Next.js Middleware（会话刷新） |

### 2.2 Middleware 会话管理

文件：`src/middleware.ts`

```typescript
import { type NextRequest } from "next/server";
import { updateSession } from "@/lib/supabase/middleware";

export async function middleware(request: NextRequest) {
  return await updateSession(request);
}
```

**职责：**
- 刷新 Supabase Session Cookie，确保会话持续有效
- 保护 Dashboard 路由（未认证用户重定向至 `/login`）
- 已登录用户访问认证页面（`/login`、`/register`）时重定向回 Dashboard

**匹配规则：** 排除静态资源路径（`_next/static`、`_next/image`、`favicon.ico`、图片文件）。

### 2.3 Server Action 认证

文件：`src/lib/dal/auth.ts`

```typescript
export async function getCurrentUserOrg(): Promise<string | null> {
  const supabase = await createClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) return null;

  const profile = await db.query.userProfiles.findFirst({
    where: eq(userProfiles.id, user.id),
  });
  return profile?.organizationId || null;
}
```

所有 Server Actions 通过调用 `getCurrentUserOrg()` 获取当前用户所属组织 ID。返回 `null` 时表示未认证，Action 应立即终止并返回错误。

### 2.4 认证方式

当前仅支持 **Email/Password 认证**，不启用社交登录。

Server Actions 定义在 `src/app/actions/auth.ts`：
- `signIn` — 邮箱密码登录
- `signUp` — 注册新用户
- `signOut` — 退出登录

## 3. 多租户

### 3.1 租户隔离机制

系统采用基于 `organization_id` 外键的逻辑隔离方案。所有核心业务表均包含 `organization_id` 字段，引用 `organizations` 表主键：

```typescript
// 示例：ai_employees 表
organizationId: uuid("organization_id")
  .references(() => organizations.id)
  .notNull(),
```

### 3.2 组织模型

文件：`src/db/schema/users.ts`

```typescript
export const organizations = pgTable("organizations", {
  id: uuid("id").defaultRandom().primaryKey(),
  name: text("name").notNull(),
  slug: text("slug").notNull().unique(),
  settings: jsonb("settings").$type<Record<string, unknown>>().default({}),
  createdAt: timestamp("created_at", { withTimezone: true }).defaultNow().notNull(),
  updatedAt: timestamp("updated_at", { withTimezone: true }).defaultNow().notNull(),
});
```

### 3.3 数据隔离要求

- DAL 查询函数必须以 `organizationId` 作为过滤条件
- Server Actions 通过 `getCurrentUserOrg()` 获取当前用户的 `organizationId`，并将其注入所有数据库操作
- 不同组织的数据完全隔离，不可交叉访问

## 4. 数据库架构

### 4.1 连接配置

文件：`src/db/index.ts`

```typescript
import { drizzle } from "drizzle-orm/postgres-js";
import postgres from "postgres";
import * as schema from "./schema";

const connectionString = process.env.DATABASE_URL!;
const client = postgres(connectionString, { prepare: false });
export const db = drizzle(client, { schema });
```

**关键配置：** `{ prepare: false }` 是 Supabase PgBouncer 连接池的必需参数，禁用 prepared statements 以兼容事务级连接池。

### 4.2 14 张数据表

Schema 定义位于 `src/db/schema/`，按模块分文件组织：

| # | 表名 | 文件 | 说明 |
|---|------|------|------|
| 1 | `organizations` | `users.ts` | 多租户根表 |
| 2 | `user_profiles` | `users.ts` | 用户档案，`id` 关联 Supabase `auth.users` |
| 3 | `ai_employees` | `ai-employees.ts` | 8 个预置 + 自定义 AI 员工 |
| 4 | `skills` | `skills.ts` | 技能定义，含分类、类型、版本 |
| 5 | `employee_skills` | `skills.ts` | 员工-技能多对多关联（level 0-100） |
| 6 | `teams` | `teams.ts` | 团队配置（场景、协作规则） |
| 7 | `team_members` | `teams.ts` | 混合成员（AI/Human），通过 `member_type` 区分 |
| 8 | `workflow_templates` | `workflows.ts` | 可复用工作流步骤定义（JSONB 存储步骤列表） |
| 9 | `workflow_instances` | `workflows.ts` | 运行中的工作流实例 |
| 10 | `workflow_steps` | `workflows.ts` | 单步进度（状态、进度百分比、输出） |
| 11 | `team_messages` | `messages.ts` | 团队协作消息（支持 AI/人类发送） |
| 12 | `tasks` | `tasks.ts` | 内容创作任务 |
| 13 | `knowledge_bases` | `knowledge-bases.ts` | 知识库定义 |
| 14 | `employee_knowledge_bases` | `knowledge-bases.ts` | 员工-知识库关联 |

所有 schema 文件通过 `src/db/schema/index.ts` 统一导出。

### 4.3 枚举定义

文件：`src/db/schema/enums.ts`

使用 Drizzle 的 `pgEnum` 定义 PostgreSQL 原生枚举类型：

```typescript
// 员工状态
employeeStatusEnum:    "working" | "idle" | "learning" | "reviewing"

// 权限级别（从低到高）
authorityLevelEnum:    "observer" | "advisor" | "executor" | "coordinator"

// 技能分类
skillCategoryEnum:     "perception" | "analysis" | "generation" |
                       "production" | "management" | "knowledge"

// 技能类型
skillTypeEnum:         "builtin" | "custom" | "plugin"

// 消息类型
messageTypeEnum:       "alert" | "decision_request" | "status_update" | "work_output"

// 工作流步骤状态
workflowStepStatusEnum: "completed" | "active" | "pending" | "skipped"

// 成员类型
memberTypeEnum:        "ai" | "human"
```

### 4.4 Drizzle ORM 约定

- **主键：** 所有表使用 `uuid("id").defaultRandom().primaryKey()`
- **时间戳：** 统一使用 `timestamp("created_at", { withTimezone: true }).defaultNow().notNull()` 和 `updated_at`
- **JSONB 字段：** 使用 `.$type<T>()` 提供 TypeScript 类型推导，如 `jsonb("rules").$type<{ approvalRequired: boolean; ... }>()`
- **级联删除：** 关联表通过 `{ onDelete: "cascade" }` 配置级联删除
- **Relations：** 每个 schema 文件导出对应的 `relations()` 定义，供 Drizzle 关系查询 API 使用
- **迁移输出：** `supabase/migrations/` 目录

## 7. Mock→DB 迁移模式

### 7.1 迁移策略

系统从 Mock 数据逐步迁移至真实数据库。Mock 数据存放在 `src/data/` 目录（19 个文件），未迁移的页面直接导入 mock 数据渲染。

### 7.2 标准迁移步骤

每个页面/模块从 mock 迁移至 DB 遵循以下六步流程：

```
步骤 1: 定义/扩展 DB Schema
  → 在 src/db/schema/ 中添加或修改表定义
  → 运行 npm run db:push 同步至 Supabase

步骤 2: 创建 DAL 查询函数
  → 在 src/lib/dal/ 中创建对应文件
  → 函数签名接收 organizationId 参数
  → 返回值类型匹配 src/lib/types.ts 中的 UI 接口
  → 负责 snake_case → camelCase 字段映射

步骤 3: 创建 Server Actions
  → 在 src/app/actions/ 中创建对应文件
  → 标记 "use server"
  → 调用 getCurrentUserOrg() 进行认证
  → 完成数据变更后调用 revalidatePath()

步骤 4: 重构 page.tsx
  → 将 import { mockData } from "@/data/..."
     替换为 const data = await dalFunction(orgId)
  → 保持 Server Component 模式

步骤 5: 移除 Mock 数据导入
  → 清理 src/data/ 中不再使用的 mock 文件

步骤 6: 同步 Schema
  → npm run db:push（开发环境直接推送）
  → npm run db:generate + npm run db:migrate（生产环境通过迁移）
```

### 7.3 DAL 函数编写规范

```typescript
// src/lib/dal/employees.ts 示例
import { db } from "@/db";
import { aiEmployees, employeeSkills, skills } from "@/db/schema";
import { eq } from "drizzle-orm";
import type { AIEmployee } from "@/lib/types";

export async function getEmployees(organizationId: string): Promise<AIEmployee[]> {
  const rows = await db.query.aiEmployees.findMany({
    where: eq(aiEmployees.organizationId, organizationId),
    with: { /* 关系查询 */ },
  });

  // DB 行 → UI 类型转换
  return rows.map(row => ({
    id: row.slug as EmployeeId,
    name: row.name,
    // ... 字段映射
  }));
}
```

```typescript
KnowledgeSource     // 知识库来源
ReviveRecommendation // 资产盘活推荐
```

### 8.3 DB 类型

文件：`src/db/types.ts`

通过 Drizzle 的 `InferSelectModel` 和 `InferInsertModel` 自动从 schema 定义派生：

```typescript
// 查询结果类型（Select）
export type AIEmployeeRow = InferSelectModel<typeof aiEmployees>;
export type TeamRow = InferSelectModel<typeof teams>;
export type SkillRow = InferSelectModel<typeof skills>;
// ... 14 张表对应 14 个 Select 类型

// 插入类型（Insert）
export type NewAIEmployee = InferInsertModel<typeof aiEmployees>;
export type NewTeam = InferInsertModel<typeof teams>;
export type NewSkill = InferInsertModel<typeof skills>;
// ... 对应 Insert 类型
```

### 8.4 类型转换规范

DAL 函数负责在 DB 类型与 UI 类型之间进行转换：

```
DB Row (snake_case)          UI Type (camelCase)
─────────────────────        ─────────────────────
organization_id        →     organizationId
avg_response_time      →     avgResponseTime
is_preset              →     isPreset
work_preferences       →     workPreferences
tasks_completed        →     tasksCompleted
```

**注意事项：**
- Drizzle ORM 在 schema 定义中已通过字段名（TypeScript camelCase）和列名（SQL snake_case）的映射处理了基础转换
- DAL 函数需额外处理嵌套对象、关联数据合并、计算字段等逻辑
- UI 类型可能包含 DB 中不存在的派生字段（如从 `EMPLOYEE_META` 获取的 `icon`、`color` 等）
