## 1. 总览：三层组装关系

技能、AI员工、团队是一个**从底到上的组装关系**，类似于「零件 → 机器人 → 车间」：

```
┌──────────────────────────────────────────────────────────────┐
│  第三层：团队 (Team)                                          │
│  = 多个AI员工 + 协作规则 + 审批流程 + 工作流                    │
│  价值：流程串联、质量管控、权限隔离、自动化生产线                  │
│  场景：多人协作完成一个完整的内容生产任务                         │
├──────────────────────────────────────────────────────────────┤
│  第二层：AI员工 (AI Employee)                                  │
│  = 角色身份 + 多个技能 + 记忆 + 知识库 + 权限等级 + 工作偏好      │
│  价值：专业化身份、经验积累、能力边界控制                         │
│  场景：执行工作流中的单个步骤，或独立完成单项任务                   │
├──────────────────────────────────────────────────────────────┤
│  第一层：技能 (Skill)                                          │
│  = 一个原子能力（搜索、写作、分析、审核...）                      │
│  价值：可复用、可组合、可独立升级的最小能力单元                    │
│  场景：AI员工执行任务时调用的具体工具                             │
└──────────────────────────────────────────────────────────────┘
```

**一句话总结：技能是"能力"，员工是"人"，团队是"组织"。**

你在技能市场上找到合适的能力，装备给合适的AI员工，然后把这些员工组成团队，让他们按照工作流协作完成任务。

## 2. 第一层：技能（Skill）

### 2.1 概念定义

技能是系统中**最小的、可复用的能力模块**。每个技能代表 AI 可以调用的一个具体工具/函数。

技能本身不能独立运行——它必须被绑定到一个AI员工身上，才能在工作流中被调用。

### 2.2 技能的核心属性

| 属性 | 说明 |
|------|------|
| 名称 / slug | 唯一标识（如 `web_search`），同时作为 LLM 工具调用的函数名 |
| 分类 | 6大类：感知、分析、生成、制作、管理、知识 |
| 类型 | `builtin`（内置）、`custom`（自定义）、`plugin`（外部API） |
| 版本 | 语义化版本号，支持版本历史和回滚 |
| 输入/输出 Schema | 定义技能的输入输出结构 |
| 运行时配置 | 平均延迟、最大并发、模型依赖等 |
| 兼容角色 | 限制哪些角色类型可以绑定此技能（空=不限） |
| 插件配置 | 仅 `plugin` 类型：HTTP端点、认证方式、请求模板等 |

#### 感知类（4个）—— 信息获取

| `news_aggregation` | 新闻聚合 | 200+新闻源、语义去重、优先级排序 |

#### 分析类（6个）—— 信息处理

| slug | 名称 | 说明 |
|------|------|------|
| `sentiment_analysis` | 情感分析 | 三级分类（正/负/中性）、句子级分析 |
| `topic_extraction` | 主题提取 | TF-IDF & TextRank、主题聚类、实体识别 |
| `competitor_analysis` | 竞品分析 | 内容对标、数据对比、策略洞察 |

| `data_report` | 数据报告 | 自动数据聚合、可视化、趋势分析 |

### 2.4 技能管理的适用场景

| 场景 | 操作 | 举例 |
|------|------|------|
| 扩展AI能力 | 创建新技能 | 接入一个外部SEO分析API作为 `plugin` 类型技能 |
| 能力市场浏览 | 查看技能列表 | 浏览所有可用技能，查看使用统计和绑定情况 |
| 技能共享 | 导入/导出 | 把自定义技能打包成 `.zip` 分享给其他组织 |
| 技能调优 | 修改/回滚 | 调整技能的输入输出定义、版本管理、回滚到历史版本 |
| 技能推荐 | 智能匹配 | 系统根据员工角色自动推荐兼容技能（按角色匹配+40分，新分类+20分） |

### 2.5 技能的三种类型

```
┌─────────────────────────────────────────────────────┐
│  builtin（内置技能）                                  │
│  系统预装的 28 个技能，覆盖内容生产全流程              │
│  直接映射到 tool-registry 中的工具函数                │
├─────────────────────────────────────────────────────┤
│  custom（自定义技能）                                 │
│  用户自行创建的技能，可以通过 .zip 包导入              │
│  包含 SKILL.md 文档 + 可选的参考/脚本文件             │
├─────────────────────────────────────────────────────┤
│  plugin（插件技能）                                   │
│  对接外部 REST API 的技能                             │
│  配置：endpoint、method、headers、auth、请求模板       │
│  系统自动生成 HTTP 调用工具                           │
└─────────────────────────────────────────────────────┘
```

## 3. 第二层：AI员工（AI Employee）

### 3.1 概念定义

AI员工是一个**有角色定位、有能力边界、有记忆积累**的持久化智能体配置。它不是简单的"技能集合"，而是一个完整的"数字人格"。

当工作流需要执行某个步骤时，系统会把AI员工的所有配置（技能、记忆、知识库、工作偏好、权限等级等）**组装**成一个完整的 LLM Agent，然后执行任务。

### 3.2 AI员工比「技能集合」多了什么

| 维度 | 技能没有的 | AI员工拥有的 |
|------|-----------|-------------|
| **角色身份** | 技能没有身份概念 | 名字、昵称、头衔、座右铭、角色类型 |
| **权限等级** | 技能不管权限 | 4级权限控制可调用的工具范围 |
| **熟练度** | 技能只有定义 | 每个技能有0-100的熟练度，影响LLM行为策略 |
| **记忆** | 技能没有记忆 | 积累历史反馈和工作模式，影响后续决策 |
| **知识库** | 技能不关联知识 | 可关联多个知识库，注入领域背景知识 |
| **工作偏好** | 技能没有风格 | 主动性、汇报频率、自主度、沟通风格 |
| **性能数据** | 技能只有使用记录 | 完成任务数、准确率、响应时间、满意度——持续追踪成长 |
| **学习能力** | 技能不会学习 | 被拒绝的模式自动记录，下次避免同样错误 |

### 3.4 权限等级体系

AI员工的权限等级（`authorityLevel`）是控制能力边界的关键机制，一个字段同时控制三个行为：

| 权限等级 | 可用工具 | 输出处理 | 适用场景 |
|----------|---------|---------|---------|
| `observer` | 无工具 | 强制需审批 | 仅观察，不参与执行 |
| `advisor` | 仅14个只读工具 | 强制需审批 | 提供建议，不直接操作 |
| `executor` | 所有绑定工具 | 按团队规则决定 | 标准执行者 |
| `coordinator` | 所有绑定工具 | 按团队规则决定 | 全权限协调者 |

**只读工具清单（advisor可用）：**
`web_search`, `trend_monitor`, `social_listening`, `news_aggregation`, `knowledge_retrieval`, `media_search`, `case_reference`, `data_report`, `sentiment_analysis`, `topic_extraction`, `competitor_analysis`, `audience_analysis`, `fact_check`, `heat_scoring`

### 3.5 技能熟练度与LLM行为

每个AI员工对每个技能有独立的熟练度（0-100），系统计算所有技能的平均熟练度，影响 LLM 的行为策略：

| 平均熟练度 | 系统提示词策略 | 行为表现 |
|-----------|--------------|---------|
| ≤30（新手） | "严格按步骤执行，避免即兴发挥" | 保守、遵循模板 |
| 31-70（熟练） | "适度自由发挥，关键决策需说明理由" | 平衡稳定与创意 |
| >70（专家） | "鼓励创新和独特见解" | 主动创新、风格鲜明 |

**熟练度自动增长机制：** 每次工作流步骤完成后，根据质量分数自动调整：

| 质量分数 | 熟练度变化 |
|---------|-----------|
| ≥ 90 | +2 |
| ≥ 80 | +1 |
| < 60 | -1 |

### 3.6 记忆系统

AI员工拥有持久化记忆，分为三种类型：

| 记忆类型 | 来源 | 重要度 | 说明 |
|---------|------|--------|------|
| `feedback` | 步骤被拒绝时 | 0.8（高） | "审核反馈：文章缺少数据支撑，需增加统计数据" |
| `pattern` | 工作流成功完成时 | 0.5（中） | "参与了'突发新闻'场景的热点监控步骤" |
| `preference` | 用户手动设置 | 自定义 | "偏好使用更口语化的标题风格" |

Agent 组装时加载重要度最高的 **top-10 条记忆**，注入到系统提示词中，让AI员工能从历史经验中学习。

### 3.7 工作偏好模板

系统提供4种预设工作偏好模板：

| 模板 | 主动性 | 汇报频率 | 自主度 | 沟通风格 | 工作时间 |
|------|--------|---------|--------|---------|---------|
| **高度自主** | 激进 | 每日 | 90% | 简洁 | 24/7 |
| **严格审批** | 保守 | 实时 | 20% | 详细 | 工作时间 |
| **均衡模式** | 平衡 | 每小时 | 60% | 标准 | 工作时间 |
| **创意优先** | 激进 | 每小时 | 75% | 创意 | 弹性 |

### 3.8 AI员工管理的适用场景

| 场景 | 操作 | 举例 |
|------|------|------|
| 调整员工能力 | 绑定/解绑技能 | 给小文绑定 `translation` 技能，使其支持多语言写作 |
| 能力边界控制 | 设置权限等级 | 将小审设为 `advisor`，确保审核时只读不改 |
| 创建新角色 | 新建员工 | 创建"社区运营官"角色，绑定 `social_listening` + `content_generate` |
| 成长追踪 | 查看详情页 | 观察小文的写作技能熟练度从50提升到85的过程 |
| 配置共享 | 导入/导出 | 把调教好的员工配置导出 JSON，在其他组织导入使用 |
| 临时停用 | 禁用/启用 | 暂时禁用小剪（视频制片人），在不需要视频产出的期间 |

## 4. 第三层：团队（Team）

### 4.1 概念定义

团队是由多个AI员工（和可选的人类成员）组成的**协作执行单元**，绑定一个工作流模板，用于完成一个完整的内容生产任务。

**核心价值：** 员工能独立干活，但团队解决的是「多人如何协作」的问题——流程串联、数据传递、审批管控、质量保障。

### 4.2 团队比「员工集合」多了什么

| 维度 | 单独使用员工 | 组建团队 |
|------|------------|---------|
| **流程串联** | 你手动指派每个员工 | 系统按工作流自动串联，上一步输出自动传给下一步 |
| **数据传递** | 你手动复制粘贴结果 | `previousSteps` 机制自动传递上游产出物 |
| **审批流程** | 没有审批，结果直接给你 | 可配置哪些步骤需要人工审批 |
| **质量管控** | 质量靠你自己判断 | 质量分低于阈值自动触发升级策略 |
| **敏感话题** | 没有全局保护 | `sensitiveTopics` 注入到每个员工的系统提示词 |
| **角色隔离** | 无法实现"自己审自己" | 不同员工独立角色，互相审核保证客观性 |
| **超时策略** | 没有超时处理 | 可配置超时后自动通过/拒绝/上报 |

### 4.3 四种预设团队场景

| 场景 | 推荐成员 | 工作流步骤 | 特点 |
|------|---------|-----------|------|
| **突发新闻** | 小雷→小策→小文→小审→小发（5人） | 监控→策划→创作→审核→发布 | 快速响应，15分钟出稿 |
| **深度报道** | 小雷→小策→小资→小文→小审→小数（6人） | 监控→策划→素材→创作→审核→分析 | 高质量长文，数据驱动 |
| **社交媒体** | 全部8人 | 完整8步流水线 | 全渠道内容生产+分发+数据闭环 |
| **自定义** | 自由选择 | 自定义流程 | 按需灵活组合 |

### 4.4 团队规则配置

```json
{
  "approvalRequired": true,           // 是否开启审批
  "reportFrequency": "realtime",      // 汇报频率
  "sensitiveTopics": ["政治", "宗教"], // 敏感话题（注入所有员工提示词）
  "approvalSteps": ["create", "publish"] // 哪些步骤需要审批
}
```

### 4.5 升级策略配置

```json
{
  "qualityThreshold": 80,        // 质量分低于此值强制审批
  "sensitivityThreshold": 0.7,   // 敏感度阈值
  "timeoutAction": "escalate",   // 超时策略：auto_approve | auto_reject | escalate
  "escalateToUserId": "xxx"      // 上报给谁
}
```

### 4.6 审批流程详解

```
步骤完成
  │
  ├── 质量分 < 60？ ──→ 强制审批
  │
  ├── 质量分 < escalationPolicy.qualityThreshold？ ──→ 强制审批
  │
  ├── 该步骤在 approvalSteps 列表中？ ──→ 需要审批
  │
  └── 以上都不满足 ──→ 自动通过，继续下一步

审批等待（最长24小时）
  │
  ├── 通过 ──→ 继续执行下一步
  │
  ├── 拒绝（第1次）──→ 反馈写入员工记忆 → 带反馈重新执行
  │
  ├── 拒绝（第2次）──→ 终止工作流
  │
  └── 超时 ──→ 按 timeoutAction 处理
       ├── auto_approve → 自动通过
       ├── auto_reject  → 标记失败
       └── escalate     → 上报给指定人员
```

### 4.7 "同一员工不同团队"的场景

同一个小文（内容创作师），可以同时属于多个团队：

| 团队 | 成员 | 规则 | 效果 |
|------|------|------|------|
| "突发新闻"团队 | 5人精简 | 无审批、无敏感限制 | 小文快速出稿，追求速度 |
| "深度报道"团队 | 6人完整 | 严格审批、敏感话题保护 | 小文仔细打磨，追求质量 |
| "品牌内容"团队 | 3人定制 | 品牌合规审批 | 小文遵循品牌调性 |

小文的技能和能力没变，但在不同团队里受到不同规则约束——就像同一个人在不同项目组里，工作方式和汇报流程是不一样的。

### 4.8 团队管理的适用场景

| 场景 | 操作 | 举例 |
|------|------|------|
| 端到端内容生产 | 选场景 + 组建团队 + 启动工作流 | "抖音热点→选题→写作→审核→发布"全自动 |
| 定制化协作流程 | 自定义场景 | 只需要"搜索→写作"两步的轻量团队 |
| 质量和风险管控 | 配置规则和升级策略 | 涉及敏感话题的内容必须人工审核 |
| 混合人机协作 | 加入人类成员 | 在审核步骤加入人类编辑作为最终把关 |
| 自动化生产线 | 绑定定时触发 | 每日6点自动执行新闻简报工作流 |

### 4.9 预设自动化工作流模板

| 模板 | 步骤 | 审批 | 触发 |
|------|------|------|------|
| **快讯自动推送** | 监控→创作→审核→发布 | 无需审批 | 事件触发 |
| **赛事速报** | 监控→创作→制作→发布 | 无需审批 | 事件触发 |
| **每日新闻简报** | 监控→策划→创作→审核→发布 | 需要审批 | 每日6:00 |
| **周度深度报道** | 完整8步 | 需要审批 | 每周一9:00 |

## 5. 三者的运行时协作流程

### 5.1 完整执行流程图

```
用户操作                     系统处理
─────────                   ─────────

① 创建/选择技能              技能入库（skills表）
       │
       ▼
② 绑定技能到员工             employee_skills关联
   设置权限/偏好              （含熟练度level、绑定类型）
       │
       ▼
③ 选场景组建团队              teams + team_members
   配置规则/审批              （含协作规则、升级策略）
       │
       ▼
④ 启动工作流                 触发 workflow/started 事件
       │
       ▼
⑤ Inngest 工作流引擎逐步执行：
       │
       ├── 步骤1: 热点监控
       │   └── 组装「小雷」为 Agent
       │       ├── 加载小雷的 4 个核心技能 → 转为可调用工具
       │       ├── 加载小雷的记忆（top-10，按重要度排序）
       │       ├── 加载关联的知识库
       │       ├── 根据权限等级过滤可用工具
       │       ├── 计算平均熟练度 → 确定行为策略
       │       ├── 注入团队的敏感话题规则
       │       └── 构建7层系统提示词 → 调用 LLM
       │           └── LLM 调用 web_search、trend_monitor 等工具
       │           └── 返回：热点列表（3-5条 + 热度评分）
       │
       ├── 步骤2: 选题策划
       │   └── 组装「小策」为 Agent
       │       ├── 自动接收步骤1的输出（热点列表）作为输入
       │       └── ...类似组装过程
       │       └── 返回：选题方案（角度、大纲、受众分析）
       │
       ├── ... 后续步骤（素材→创作→制作→审核→发布→分析）
       │
       └── 每个步骤完成后的自动处理：
           ├── 更新员工技能熟练度（质量高→+2，低→-1）
           ├── 记录技能使用数据（用于统计分析）
           ├── 更新员工性能指标（任务数、准确率等）
           ├── 检查是否需要人工审批（团队规则 + 质量阈值）
```

### 5.2 Agent 组装的 7 层系统提示词

当一个AI员工被激活执行步骤时，系统构建一个由7层组成的系统提示词：

```
┌─────────────────────────────────────────┐
│ Layer 1: 身份层                          │
│ "你是{昵称}，{角色名}，座右铭是..."        │
├─────────────────────────────────────────┤
│ Layer 2: 能力层 + 熟练度策略              │
│ "你拥有以下技能：[工具列表及描述]"          │
│ "你的平均熟练度为{N}，[行为策略指导]"       │
├─────────────────────────────────────────┤
│ Layer 3: 权限层                          │
│ "你的权限等级为{executor}，可执行以下操作" │
│ "超出权限范围的操作需上报"                 │
├─────────────────────────────────────────┤
│ Layer 3.5: 敏感话题层（来自团队规则）      │
│ "以下话题需特别注意：[政治, 宗教, ...]"    │
│ "涉及这些话题时需人工审核"                 │
├─────────────────────────────────────────┤
│ Layer 4: 知识层                          │
│ "你可参考以下知识库：[名称及描述列表]"      │
├─────────────────────────────────────────┤
│ Layer 5: 工作风格层                       │
│ "主动性：{proactivity}，自主度：{N}%"     │
│ "沟通风格：{style}，工作时间：{hours}"     │
├─────────────────────────────────────────┤
│ Layer 6: 经验记忆层                       │
│ "[feedback] 上次文章因缺少数据被拒..."      │
│ "[pattern] 成功完成过社交媒体场景..."       │
├─────────────────────────────────────────┤
│ Layer 7: 输出规范 + 质量自评               │
│ "使用中文输出，结构化 Markdown 格式"        │
│ "质量自评：完整性30%+准确性30%+创意20%+格式20%" │
└─────────────────────────────────────────┘
```

### 5.3 步骤间数据传递机制

工作流引擎通过 `previousSteps` 机制自动传递上游产出：

```
步骤1 (小雷) 输出：
  ├── summary: "发现3个高热度话题：..."

         │
         ▼ 自动注入到步骤2的用户消息中
步骤2 (小策) 收到的上下文：
  ├── ## 上游步骤：monitor（小雷）
  │   摘要：发现3个高热度话题...
  │   产出物：
  │     - [topic_brief] 热点分析报告
  └── ## monitor 输出详情
      ### 热点分析报告 (topic_brief)
      {完整内容}
```

### 6.3 什么时候不需要团队？

如果你的业务场景是以下情况，**直接用AI员工即可，不必组建团队**：

- 只需要一个AI完成一件事（写文章、搜热点、分析数据）
- 不需要多步骤串联
- 不需要审批流程
- 不需要敏感话题保护

## 7. 常见问题

### Q1：把所有技能绑在一个AI员工上，这个员工不是什么都能做了吗？为什么还需要多个员工？

技术上可以，但会导致以下问题：

| 问题 | 原因 |
|------|------|
| **系统提示词臃肿** | 28个技能的描述全部注入，LLM注意力分散，每个技能执行质量下降 |
| **模型选择被锁定** | 系统根据第一个技能分类选模型（感知→GPT-4o-mini，生成→Claude），全能员工只能命中一个模型 |
| **熟练度被平均化** | 28个技能取平均值趋向中间值（~50），失去"专家级"行为指导的优势 |
| **记忆混杂** | 写作反馈和审核反馈混在一起，top-10记忆可能加载到不相关的内容 |
| **自己审自己** | 同一个系统提示词和记忆，审核步骤无法客观评判自己产出的内容 |
| **权限无法隔离** | 一个员工只有一个权限等级，要么全能（失去安全边界）要么受限（失去全能的意义） |

**专业分工的核心优势：**

| 维度 | 全能员工 | 专业员工 |
|------|---------|---------|
| 系统提示词 | 1套（泛泛而谈） | 每人1套（精准专业） |
| LLM模型 | 1个（不一定最优） | 每人匹配最优模型 |
| 审核客观性 | 自己审自己 | 独立角色互审 |
| 权限控制 | 无法隔离 | 最小权限原则 |
| 记忆积累 | 混杂 | 专业领域聚焦 |
| 熟练度成长 | 趋向平均 | 专业技能持续提升 |

### Q2：有了员工之后，为什么还需要团队？

员工解决"谁来做"，团队解决"怎么配合做"。

**不需要团队的场景：**
> "帮我把这篇文章改写成小红书风格" → 直接让小文执行

**需要团队的场景：**
> "抖音上某话题很火，帮我从热点分析到内容发布全流程跑一遍"

这需要5个员工按顺序配合，数据在步骤间自动传递，关键步骤卡审批——这就是团队做的事。

### Q3：团队规则中的敏感话题是怎么生效的？

团队规则中设置的 `sensitiveTopics`（如 `["政治", "宗教"]`）会在 Agent 组装时注入到该团队里**每一个员工**的系统提示词 Layer 3.5 中，确保整条生产线都规避这些话题。不同团队可以有不同的敏感话题配置。

### Q4：技能的 `compatibleRoles` 有什么作用？

这是一个安全机制。如果技能设置了 `compatibleRoles: ["content_writer", "quality_reviewer"]`，那只有角色类型为 `content_writer` 或 `quality_reviewer` 的员工才能绑定这个技能。空数组表示不限制。

绑定时系统会检查兼容性，不兼容的绑定会被拒绝。

## 8. 数据模型参考

### 8.1 核心表关系

```
skills ──────┐
  (28+个技能)  │  employee_skills（多对多）
             ├──── level: 0-100（熟练度）
             │     bindingType: core | extended | knowledge
ai_employees ┘
  (8+个员工)  ─┐
               │  team_members（多对多）
               ├──── teamRole: "热点发现"
               │     memberType: ai | human
teams ────────┘
  (N个团队)   ─── rules（协作规则）
              ─── escalationPolicy（升级策略）
              ─── workflowTemplateId → workflow_templates

workflow_templates ─── workflow_instances（执行实例）
                      ─── workflow_steps（执行步骤）
                          ─── workflow_artifacts（产出物）
                          ─── execution_logs（执行日志）
                          ─── skill_usage_records（技能使用记录）
```

### 8.2 辅助表

| 表 | 说明 |
|----|------|
| `skill_versions` | 技能版本历史，支持回滚 |
| `skill_files` | 技能附件（参考文档 + 脚本） |
| `skill_combos` | 技能组合（预定义技能管线，待实现） |
| `skill_usage_records` | 技能使用统计（成功率、质量分、耗时、token） |
| `employee_memories` | 员工记忆（反馈、模式、偏好） |
| `employee_knowledge_bases` | 员工-知识库关联 |
| `employee_versions` | 员工配置版本历史 |

### 8.3 关键枚举

| 枚举 | 值 |
|------|---|
| `skillCategory` | perception, analysis, generation, production, management, knowledge |
| `skillType` | builtin, custom, plugin |
| `skillBindingType` | core, extended, knowledge |
| `authorityLevel` | observer, advisor, executor, coordinator |
| `employeeStatus` | working, idle, learning, reviewing |
| `memberType` | ai, human |
