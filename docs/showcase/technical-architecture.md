### 1.2 各层职责与层间关系

**基础设施层**：提供数据库存储、大语言模型调用、向量嵌入计算、后台任务调度等基础能力。这一层的组件都是可替换的——今天用 DeepSeek，明天换 Claude 或 GPT，只需要改配置，上层无感知。

**能力基座层**：这是系统的「零件库」。技能、知识库、记忆、工具都是独立的、可组合的原子能力。它们不直接面向用户，而是被上层的智能体调用。核心设计原则是**「模块化 + 可组合」**——任何一个新的 AI 员工或工作流，都是从这一层挑选零件组装出来的。

**智能体层**：这是系统的核心。包含两种形态的智能体——AI 员工（垂类深度智能体）和工作流（灵活编排引擎），以及统一的对话中心作为交互入口。AI 员工和工作流之间不是对立关系，而是**互补协作**的：AI 员工可以被工作流编排调用，工作流的每个步骤本质上就是调用某个 AI 员工的某个技能。

**业务应用层**：面向用户的具体功能页面。每个应用模块不直接调用 LLM，而是通过智能体层间接驱动。这使得新增业务应用时，只需要对接智能体层的接口，不需要重新处理 AI 调用逻辑。

## 二、核心设计哲学

### 2.1 为什么需要「技能」这个抽象层？

在没有技能系统的设计中，AI 的能力直接写在 Prompt 里。这会导致三个问题：

| 问题 | 没有技能系统 | 有技能系统 |
|------|-------------|-----------|
| 能力复用 | 两个员工都需要"搜索"能力，Prompt 写两遍，改了一处忘改另一处 | 技能是独立模块，多个员工共享同一个搜索技能 |
| 能力量化 | 不知道员工哪项能力强哪项弱，无法优化任务分配 | 每个技能有熟练度评分，任务分配时优先匹配高熟练度员工 |
| 能力扩展 | 新增能力要改 Prompt，改完还要回归测试全部场景 | 新增一个技能模块，绑定到需要的员工上，其他不受影响 |

**技能系统的设计**：

- **技能 = 能力的原子单元**。每个技能包含：名称、描述、执行指南（Prompt 模板）、绑定的工具集、权限等级。
- **多对多绑定**：一个技能可以被多个员工拥有（如"全网搜索"同时属于热点猎手和数据分析师），一个员工可以拥有多个技能（热点猎手同时有搜索、热度分析、社交监听等技能）。
- **权限等级**：每个技能对应三种权限——观察者（只读分析）、顾问（建议但不执行）、领导者（完全执行权）。这限定了员工能调用哪些工具。
- **熟练度评分**：基于历史执行数据和用户反馈，自动计算每个员工在每个技能上的熟练度（0-100），影响 AI 执行行为和自主判断空间。

#### 技能熟练度评分规则

每次技能执行后，系统根据 VerifyLearner 的质量评分自动更新熟练度：

| 质量评分 | 熟练度变化 | 计算规则 |
|---------|-----------|---------|
| ≥ 80 分 | +3 ~ +5 | 基础 +3，每超过 80 分 10 分额外 +1（如 95 分 → +4） |
| 60-79 分 | 不变 | 合格但无突出表现，维持现状 |
| < 60 分 | -2 | 质量不达标，熟练度下降 |

熟练度存储在 `employee_skills.level` 字段，通过原子 SQL 操作更新，范围锁定在 0-100 之间。同时更新使用次数（`usageCount`）和成功次数（`successCount`）。

**熟练度对行为的影响**：在 Agent Assembly 时，系统计算员工所有技能的平均熟练度，注入 Prompt 中作为行为指导：

| 平均熟练度 | 行为指导 |
|-----------|---------|
| ≤ 30（新手）| "请严格按步骤执行，每步自检，避免跳步或发挥" |
| 31-70（中级）| "可以适当发挥，但关键判断需说明依据" |
| > 70（专家）| "可以自由发挥，鼓励创新和独到见解" |

> **实现说明**：目前 Leader Agent 在规划任务、分配员工时，只能看到员工的技能名称列表，尚未接收熟练度数值。熟练度主要影响执行时的行为自由度。后续规划将把熟练度引入 Leader 的员工匹配决策。

### 2.2 AI 员工与工作流：两种智能体形态

这是系统架构中最重要的设计决策之一。AI 员工和工作流不是并列的两个独立功能，而是**同一套智能体能力的两种使用形态**。

#### AI 员工：垂类场景的深度智能体

AI 员工的本质是**为特定垂类场景预设好的专家型智能体**。以「热点猎手（小雷）」为例：

- **预设的技能组合**：全网搜索 + 社交监听 + 热度分析 + 趋势评分
- **绑定的知识库**：新闻行业规范、平台热度算法、历史热点数据
- **持续积累的记忆**：哪些类型的热点用户感兴趣、哪些来源可靠、过去哪些判断是错的
- **独特的人格和风格**：敏锐、快速、注重时效性

这使得用户可以直接和小雷对话——"帮我看看今天有什么热点"——不需要配置任何流程，小雷自带了完整的热点捕获能力。**从某种意义上说，每个 AI 员工就是一个「预设好的单步工作流」，专注于一个垂直场景的深度耕耘。**

#### 工作流：灵活的流程编排引擎

工作流解决的是**跨职能协作**的问题。当任务复杂到需要多个 AI 员工配合时，就需要工作流来编排：

- **步骤序列定义**：工作流模板定义了一系列步骤，每个步骤只指定需要哪个技能（`skillName`），**不预设由哪个员工执行**——具体的员工分配由 Leader Agent 在运行时根据技能匹配动态决定
- **DAG 依赖管理**：步骤之间可以有依赖关系（先搜索再分析、先写稿再审核），支持串行和并行
- **场景自定义**：用户可以根据自己的业务场景自定义工作流模板，复用模板快速启动
- **数据传递**：上一步的输出自动成为下一步的输入上下文

#### 两者的关系和使用场景

**什么时候用 AI 员工直接对话？**
- 任务属于单一领域（只需要一个专家）
- 需要多轮交互式探索（"这个热点再深入分析一下"）
- 即时性需求（突发事件，不想配置流程，直接找人问）
- 日常高频操作（每天早上让小雷看热点、让小数出报表）

**什么时候用工作流？**
- 任务涉及多个专业领域（热点发现→选题→写稿→审核→发布）
- 流程标准化、可复用（每周的选题会流程、日常发稿流程）
- 需要审批节点和过程追踪
- 需要步骤间的数据自动传递

**关键区别：**
- AI 员工侧重**深度**——在垂类场景下越用越专业（知识库+记忆+学习）
- 工作流侧重**广度**——跨职能灵活编排、场景可定制
- 两者可以嵌套：工作流的每个步骤调用 AI 员工执行，AI 员工在工作流中也能积累经验

## 三、知识库体系

### 3.1 为什么需要知识库？

大语言模型有三个天然局限：
1. **知识截止日期**——不知道最近发生了什么
2. **没有领域深度**——通用知识很广，但在特定行业（如某个地方媒体的报道规范）不够深入
3. **Prompt 长度有限**——无法把全部领域知识塞进一次对话

知识库通过 **RAG（检索增强生成）** 模式解决这三个问题：把领域知识持久化存储，执行时只检索最相关的片段注入上下文，让通用大模型具备领域专业度。

│             │                                          │  知识库 C    │
│  内容创作师  │──── 绑定 ──►  写作规范知识库                │  ...        │
│  (小文)     │──── 绑定 ──►  品牌调性知识库                │             │
│             │                                          │             │
└─────────────┘                                          └─────────────┘

关键规则：
  • 员工只能检索已绑定的知识库（权限隔离）
  • 同一知识库可被多个员工共享
  • 只有 vectorization_status = 'done' 的知识库才可检索
```

**执行时的检索流程：**

1. **Agent Assembly 阶段**：加载该员工绑定的所有知识库，注册 `kb_search` 工具
2. **LLM 执行阶段**：AI 员工在思考过程中，自主决定是否需要检索知识库、检索什么关键词
3. **检索返回**：系统对查询向量化 → 与知识库向量做余弦相似度计算 → 返回 Top-K 最相关片段
4. **上下文注入**：检索结果作为工具调用结果返回给 LLM，辅助生成更专业的回答

这个设计的关键点是：**检索时机由 AI 自主决定**，而非硬编码"每次都检索"。这确保了只在需要领域知识时才消耗检索资源，同时让 AI 能够根据问题的复杂度决定检索的深度。

## 四、认知进化闭环

这是整个系统最核心的设计——让 AI 员工不是静态的 Prompt 机器人，而是**能持续进化的智能体**。每次任务执行都在积累经验、优化行为，形成一个持续旋转的飞轮。

> **贯穿示例**：为了让每个环节的讲解更具象，本章以一个真实使用场景贯穿所有环节——

### 4.2 ① 意图理解：自然语言到精准路由

用户在对话中心输入任意自然语言。意图识别引擎的工作过程：

**构建分析上下文**：动态加载当前可用的技能目录、8个AI员工的画像和能力标签、已配置的工作流模板列表、历史对话中积累的 few-shot 示例。

**LLM 分析**：将上下文和用户输入一起发给 LLM，分析出：
- **意图类型**（8种）：技能执行、工作流触发、多步骤任务、知识检索、通用对话等
- **目标员工**：应该由哪个 AI 员工处理
- **目标技能**：调用该员工的哪个具体技能
- **如果是复杂任务**：拆解为多个子任务的计划

**校验与兜底**：对 LLM 的输出进行校验——员工 ID 是否合法、技能名是否存在、工作流是否已配置。如果解析失败，graceful fallback 到通用对话模式。

> **📌 场景示例 · 意图理解**
>
> ```
> 用户输入："今天'AI+教育'话题很火，帮我规划选题方案，生成微信公众号、抖音、微博三个平台的稿件"
>          ↓
> 意图识别引擎构建分析上下文：
>   - 加载 8 个 AI 员工画像（小雷=热点搜索, 小策=选题策划, 小文=内容创作...）
>   - 加载 28+ 技能目录（全网搜索, 热度分析, 选题策划, 内容生成, 多平台适配...）
>   - 加载已配置的工作流模板（如"热点→选题→写稿→审核"模板）
>   - 加载历史 few-shot 示例
>          ↓
> LLM 分析结果：
>   - 意图类型：multi_step_task（多步骤任务）
>   - 涉及多个员工：热点猎手(搜索)、选题策划师(策划)、内容创作师(写稿)、渠道运营师(适配)
>   - 判断：需要工作流编排，非单员工能完成
>          ↓
> 路由到 → 工作流引擎 → Leader Agent 进行任务拆解
> ```

### 4.3 ② 技能的学习与发现

**已实现 — 技能熟练度自动更新：**
- 每次技能执行后，根据自评分数和用户反馈更新该员工在该技能上的熟练度
- 高分执行 → 熟练度提升，低分执行 → 熟练度下降
- 熟练度影响任务分配：同一技能有多个员工拥有时，优先分配给熟练度最高的员工

**已实现 — 技能执行模式积累：**
- 成功的执行模式（≥80分）被提取为记忆，下次执行同类任务时自动参考
- 失败的执行模式（＜60分）被标记为"避免"模式，防止重复犯错

**近期规划 — 技能需求自动发现：**
- AI 在执行任务时发现自己缺少某项能力（如需要处理视频但没有视频分析技能）
- 系统自动记录能力缺口，汇总后推荐需要新增的技能
- 管理员审核后可一键创建新技能并绑定到员工

> **📌 场景示例 · 技能学习与发现**
>
> ```
> Leader Agent 加载所有可用员工及其技能，为任务匹配最佳人选：
>
>   小雷（热点猎手）：全网搜索(level:82), 社交监听(level:75), 热度分析(level:88)
>   小策（选题策划师）：选题策划(level:71), 角度设计(level:65), 受众分析(level:60)
>   小文（内容创作师）：内容生成(level:85), 风格改写(level:78), 标题创作(level:90)
>   小发（渠道运营师）：多平台适配(level:72), 发布策略(level:68)
>
> 匹配结果：
>   - "搜索AI教育热点" → 需要"全网搜索"+"热度分析" → 小雷（熟练度最高）
>   - "规划选题方案"   → 需要"选题策划"+"角度设计" → 小策
>   - "生成稿件内容"   → 需要"内容生成"           → 小文（level:85）
>   - "适配三平台"     → 需要"多平台适配"          → 小发
>
>   ※ 如果某个子任务找不到匹配技能（如需要"视频脚本"但无人拥有），
>     系统记录能力缺口 → 后续推荐管理员新增该技能（近期规划功能）
> ```

**关键机制：**
- **DAG 依赖调度**：无依赖的任务并行执行（如上面的子任务3和4），有依赖的顺序执行
- **员工匹配**：Leader Agent 根据子任务的技能需求和可用员工的技能列表智能匹配。工作流模板只定义技能需求（如"热度分析"），不绑定具体员工。Leader 加载所有可用员工及其技能列表，由 AI 智能判断最合适的人选。如果 Leader 分配了不存在的员工，系统自动兜底到第一个可用员工

- **中间产物传递**：上一步的输出（workflow_artifacts）自动作为下一步的输入上下文

### 4.5 ④ 智能执行：Agent Assembly 与 7 层提示词

每次 AI 员工执行任务（无论是直接对话还是工作流步骤），都会经历 **Agent Assembly（智能体组装）** 过程：

**组装步骤（并行加载）：**
1. 加载员工基础信息（身份、性格、风格）
2. 加载已绑定技能列表 + 计算熟练度
3. 加载已绑定知识库 + 注册 `kb_search` 工具
4. 加载历史记忆（按重要性排序，取 Top-10，阈值 ≥ 0.3）
5. 根据权限等级过滤可用工具集
6. 构建 7 层系统提示词

**7 层提示词架构：**

| 层级 | 名称 | 内容 | 动态程度 |
|------|------|------|---------|
| Layer 1 | 身份层 | 员工姓名、角色、性格特征、工作风格 | 静态（建员工时定义）。工作流中由 Leader 动态分配员工后确定 |
| Layer 2 | 技能层 | 已绑定技能列表 + 各技能熟练度 + 执行指南 | 半动态（熟练度随反馈变化）|
| Layer 3 | 权限层 | 权限等级约束（观察者/顾问/领导者） | 静态 |
| Layer 4 | 知识层 | 绑定的知识库清单 + kb_search 工具说明 | 动态（知识库内容持续更新）|
| Layer 5 | 风格层 | 敏感话题规避 + 工作偏好 | 半动态 |
| Layer 6 | 记忆层 | Top-10 历史记忆 + 学习模式 | **高度动态**（每次执行后都可能更新）|
| Layer 7 | 输出层 | 输出格式要求 + 质量自评评分标准 | 静态 |

**工作流中 Layer 1 如何确定？**

在直接对话模式下，用户选了和谁聊天，Layer 1 就是谁。但在工作流模式下，流程模板只定义了技能需求（如"热度分析"），没有指定具体员工。确定过程是：

```
工作流模板步骤 (只有 skillName: "热度分析")
        ↓
Leader Agent 加载所有可用员工 + 技能列表
        ↓
Leader AI 智能匹配："热度分析"技能 → 小雷最合适
        ↓
写入 missionTask.assignedEmployeeId = 小雷的ID
        ↓
执行时 assembleAgent(小雷的ID)
        ↓
Layer 1 = 小雷的身份信息（姓名、角色、性格、风格）
```

这意味着**同一个工作流模板在不同执行中，可能由不同员工来执行同一个步骤**——如果有新员工被创建且更擅长某个技能，Leader 会自动选择更优人选。

**分层设计的核心价值**：每一层可以独立更新。知识库有新文档了只影响 Layer 4；用户给了负面反馈只影响 Layer 6；新增一个技能只影响 Layer 2。不需要重写整个 Prompt，系统自动组装出当前最新状态的完整提示词。

> **📌 场景示例 · 智能执行（以任务3「小文撰写稿件」为例）**
>
> ```
> Agent Assembly 组装过程：
>
>   1. 加载小文的员工信息 → Layer 1: "你是小文，内容创作师，擅长..."
>   2. 加载技能列表 → Layer 2: "你拥有以下技能：内容生成(熟练度85/100)..."
>      平均熟练度 84 → 注入行为指导："你的技能熟练度较高，可以自由发挥"
>   3. 加载权限约束 → Layer 3: "你有顾问级权限..."
>   4. 加载绑定的知识库 → Layer 4: "写作规范知识库、品牌调性知识库"
>      注册 kb_search 工具
>   5. 加载敏感话题 → Layer 5: "避免涉及..."
>   6. 加载 Top-10 记忆 → Layer 6:
>      - [feedback] "用户上次对深度分析类稿件表示赞赏"（重要性 0.75）
>      - [failure_lesson] "上次AI教育稿件数据引用不够权威"（重要性 0.8）
>      - [success_pattern] "列举3个具体案例的写法获得高分"（重要性 0.85）
>   7. 输出格式 + 自评标准 → Layer 7
>
>   → 组装为完整的 7 层系统 Prompt
>   → 将任务2（选题方案）的输出作为用户消息传入
>   → 调用 LLM 执行（最多 20 步工具调用，3分钟超时）
>
> 执行过程中：
>   - 小文自主调用 kb_search 检索"写作规范知识库"中的深度报道模板
>   - 参考 Layer 6 记忆："上次数据引用不权威" → 这次主动引用教育部官方数据
>   - 参考记忆："列举3个案例的写法获得高分" → 采用案例列举法组织稿件
> ```

### 4.6 ⑤ 结果验证：双通道质量保障

每次执行完成后，有两个质量验证通道：

**通道一：LLM 自我评估（VerifyLearner）**

每次 AI 员工执行完任务后，系统异步（fire-and-forget，不阻塞主流程）调用一个独立的 LLM 评估器（DeepSeek-chat，temperature=0.2 确保评分稳定），对产出内容进行四维质量打分。

**评估 Prompt 结构：**
```
你是一个严格的质量评审员。请评估以下任务产出的质量。

## 任务信息
- 标题：{taskTitle}
- 描述：{taskDescription}
- 期望输出：{expectedOutput}（如有）

## 产出内容
{output}（截断到6000字符以内）

## 评估要求
请从以下维度评估，并给出 0-100 的总分：
1. 准确性 (accuracy)    — 内容是否准确、事实是否正确
2. 完整性 (completeness) — 是否完整覆盖了任务要求
3. 风格 (style)         — 格式和表达是否恰当
4. 合规性 (compliance)   — 是否符合政策和规范要求

输出 JSON：qualityScore, passed, feedback, issues[], successPattern, failureLesson
```

**评分结果的处理逻辑：**

```
LLM 返回 qualityScore (0-100) + issues[] + successPattern/failureLesson
                          │
                          ▼
          ┌───────────────────────────────────────┐
          │  ① 写入 verificationRecords 表         │ ← 完整评分记录持久化
          │     (组织ID、任务ID、评分、issues等)      │
          ├───────────────────────────────────────┤
          │  ② 根据分数生成记忆写入 employeeMemories │
          │                                       │
          │  ≥ 80 分 → 提取成功模式                │
          │    记忆重要性 = score/100（如85分→0.85） │
          │                                       │
          │  ＜ 60 分 → 提取失败教训               │
          │    记忆重要性 = 0.8（固定）              │
          │                                       │
          │  任意高严重度 issue → 提取技能洞察       │
          │    记忆重要性 = 0.7（固定）              │
          ├───────────────────────────────────────┤
          │  ③ 更新技能熟练度                       │
          │    调用 updateSkillStats()             │
          │    根据评分调整该员工各技能的 level 值     │
          └───────────────────────────────────────┘
```

**关键实现细节**：验证结果是**内联实时处理**的——`verify()` 函数内部直接写入 `verificationRecords` 和 `employeeMemories` 两张表，**不经过定时任务**，不需要额外的后台处理。同时也会同步更新技能熟练度。整个过程异步执行不阻塞用户响应。

**验证级别**：系统根据意图类型自动判断验证严格程度：
- `simple`：通用对话、信息检索
- `important`：内容创作、深度分析、数据分析、质量审核、媒体制作
- `critical`：发布操作、工作流最终步骤

---

**通道二：用户反馈**

用户反馈分为两条独立路径，存储机制和处理方式完全不同：

**路径 A：对话场景的点赞/点踩（即时记忆）**

用户在对话中点击👍/👎 → POST `/api/chat/feedback` → **直接写入 `employeeMemories` 表**，不经过中间表，不等定时任务。

```
用户操作        │  记忆内容                                              │ 重要性
─────────────────┼───────────────────────────────────────────────────────┼──────
👍 点赞         │  "用户对这条回复表示赞赏。被肯定的要点：'{摘要}'。            │  0.75
               │   请在类似场景下继续保持这种输出风格与深度。"               │
👎 点踩         │  "用户对这条回复不满意。需要避免的问题：'{摘要}'。            │  0.9
               │   请在类似场景下调整表达方式、补充信息或换用更合适的切入角度。" │
```

这些记忆在下次 Agent Assembly 时直接加载到 Layer 6，**立即影响下次执行行为**。

**路径 B：工作流场景的接受/拒绝/编辑（聚合学习）**

工作流步骤产出经人工审核后的反馈（接受/拒绝/编辑）→ 写入 `user_feedback` 表 → **由学习引擎定时聚合分析**（见4.7节）。

**负面反馈权重高于正面（0.9 > 0.75）**——这是刻意设计的。因为"避免犯错"比"重复成功"更重要：一次糟糕的输出对用户信任的伤害远大于一次优秀输出带来的提升。

> **📌 场景示例 · 结果验证**
>
> **自评通道（异步，不阻塞返回）：**
>
> ```
> 小文的稿件产出 → VerifyLearner
>
> LLM 评估器分析：
>   - 准确性: 事实引用有教育部官方数据来源 ✓
>   - 完整性: 覆盖了选题方案中的所有角度 ✓
>   - 风格: 符合深度报道的叙事风格 ✓
>   - 合规性: 无敏感内容 ✓
>
> 评分结果：qualityScore = 87
>
> 处理：
>   ① 写入 verificationRecords（完整评分记录）
>   ② 87 ≥ 80 → 提取成功模式记忆：
>      "AI教育话题采用'政策背景+案例分析+趋势展望'三段式结构获得高质量评分"
>      重要性 = 87/100 = 0.87 → 写入 employeeMemories
>   ③ 更新小文的"内容生成"技能熟练度：
>      87 ≥ 80 → delta = 3 + floor((87-80)/10) = 3
>      level: 85 → 88
> ```
>
> **用户反馈通道（稍后触发）：**
>
> ```
> 用户审阅稿件产出：
>   - 微信版本 👍 → 直接写入 employeeMemories
>     "用户对这条回复表示赞赏。被肯定的要点：'AI教育三段式结构稿件'"
>     重要性 0.75
>   - 抖音脚本 👎 → 直接写入 employeeMemories
>     "用户对这条回复不满意。需要避免的问题：'抖音脚本太长，不够口语化'"
>     重要性 0.9（负面权重更高！）
> ```

### 4.7 ⑥ 学习引擎：从经验到行为优化

学习引擎是闭环的最后一步，也是连接"过去经验"和"未来行为"的桥梁。

**两条学习路径**：系统有两种不同的学习机制，分别处理不同来源的反馈：

**路径一：即时学习（对话反馈 + 自评记忆 → 直接生效）**

对话中的点赞/点踩和 VerifyLearner 的自评结果，**直接写入 `employeeMemories` 表**，下次 Agent Assembly 时自动加载到 Layer 6。无需学习引擎介入，即时生效。

**路径二：聚合学习（工作流反馈 → 学习引擎 → 模式提取）**

工作流场景下的接受/拒绝/编辑反馈存在 `user_feedback` 表中，由学习引擎定期聚合分析。

**触发时机**：每日凌晨 02:00（Asia/Shanghai）定时运行 + 每次工作流执行完成后由事件触发。

**处理流程：**

```
┌─────────────────────────────────────────────────────────────┐
│                     学习引擎处理流程                           │
│                                                             │
│  数据来源：user_feedback 表（工作流步骤的接受/拒绝/编辑反馈）    │
│                                                             │
│  1. 按员工遍历，查询 user_feedback 表                         │
│     按 feedbackType × stepKey 分组统计                        │
│     │                                                       │
│     ▼                                                       │
│  2. 提取三类模式（达到阈值才提取）：                             │
│     │                                                       │
│     ├─ 🚫 拒绝模式（同一 stepKey 被拒绝 ≥ 2次）              │
│     │   → "避免: {stepKey} 步骤常见拒绝原因"                  │
│     │                                                       │
│     ├─ ✏️  编辑模式（同一 stepKey 被编辑 ≥ 3次）              │
│     │   → "改进: {stepKey} 步骤输出需人工调整"                │
│     │                                                       │
│     └─ ✅ 接受模式（同一 stepKey 被接受 ≥ 5次）              │
│         → "保持: {stepKey} 步骤输出质量良好"                  │
│                                                             │
│  3. 持久化：                                                 │
│     → 更新 ai_employees.learnedPatterns（JSONB 字段）         │
│       记录 pattern 来源、计数、最后出现时间                     │
│     → 生成学习引擎摘要记忆存入 employee_memories               │
│                                                             │
│  4. 闭环效果：                                               │
│     下次 Agent Assembly 时：                                  │
│     → Layer 6（记忆层）自动包含这些学习成果                      │
│     → AI 员工的行为因此发生改变                                │
│     → 无需人工调整 Prompt                                     │
└─────────────────────────────────────────────────────────────┘
```

> **📌 场景示例 · 学习引擎**
>
> ```
> 两条路径并行：
>
> 路径A（即时生效）：
>   上面产生的记忆已经直接写入 employeeMemories
>   → 下次小文执行时，Agent Assembly 加载 Top-10 记忆
>   → "抖音脚本太长不够口语化"会以 0.9 的高权重被优先加载
>   → 小文下次写抖音脚本时会自动注意精简口语化
>
> 路径B（学习引擎聚合，凌晨 02:00 或工作流完成时）：
>   假设小发的"多平台适配"步骤近期累积了 3 次编辑反馈：
>   → 学习引擎提取模式："改进: multi_platform_adapt 步骤输出需人工调整"
>   → 更新 ai_employees.learnedPatterns
>   → 生成学习摘要记忆写入 employeeMemories
> ```
>
> **飞轮转动效果**：下次用户再让系统做多平台内容时——
> - 小文写稿会参考"三段式结构获得高分"的成功记忆
> - 小发做抖音适配会参考"脚本太长不够口语化"的教训记忆
> - 两人的行为都因为这次执行而变得更好

### 4.8 闭环的飞轮效应

这个闭环不是执行一次就结束了，而是**持续旋转的飞轮**：

- **第一圈**：AI 员工基于初始配置执行任务，产出质量参差不齐
- **第二圈**：用户反馈开始积累，失败教训被记住，同类错误不再重犯
- **第三圈**：成功模式开始固化，AI 员工形成了稳定的执行风格
- **第N圈**：员工的知识库持续丰富、记忆持续积累、技能熟练度持续优化——**越用越好**

**与传统 AI 应用的根本区别**：大多数 AI 应用是无状态的——今天犯的错明天还会犯。我们的系统通过知识库 + 记忆 + 学习引擎三位一体的机制，让每个 AI 员工都具备了**有状态的持续进化能力**。

## 五、技术实现要点

### 5.1 关键技术选型与理由

| 组件 | 选型 | 选型理由 |
|------|------|---------|
| 框架 | Next.js 16 + React 19 | App Router 的 Server Component 模式天然适合 AI 应用的流式渲染 |
| 数据库 | Supabase PostgreSQL | 开源、自托管能力、内置 Auth、支持 JSONB（存向量和学习模式）|
| ORM | Drizzle 0.45 | 类型安全、轻量、迁移管理清晰 |
| AI 调用 | Vercel AI SDK v6 | 统一的 LLM 调用接口、内置工具调用、流式支持 |
| 向量化 | Jina Embeddings v3 | 1024维高精度向量、中文优化、批量处理能力 |
| 任务调度 | Inngest | 事件驱动、自带重试和超时、可视化任务流 |
| 搜索 | Tavily API | 专为 AI Agent 设计的搜索 API，返回结构化结果 |

### 5.2 多租户架构

系统设计为多租户 SaaS 架构：
- 所有核心表都有 `organization_id` 外键
- 知识库、记忆、工作流、员工配置都在组织维度隔离
- 支持组织级别的自定义配置（RBAC 角色权限系统已实现）

### 5.3 数据规模与性能考量

- **数据库**：~145 张表、72 个枚举类型，覆盖完整的业务数据模型
- **向量检索**：当前采用应用层余弦相似度计算（JSONB 存储），预留了 pgvector 升级路径（当知识块超过 ~10k 条时切换）
- **异步处理**：所有重计算（向量化、学习引擎、竞品分析）通过 Inngest 异步执行，不阻塞用户操作
- **Supabase 连接**：使用 PgBouncer 连接池 + `prepare: false` 配置，确保连接复用
