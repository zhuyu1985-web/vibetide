# 新闻研究报告 — 端到端操作手册

> **用一个具体场景把所有模块串起来：做一份《2025 上半年重庆乡村振兴报道分析》研究报告。**

> 完成本手册后你将掌握：从配置数据源 → 自动采集 → 高级检索 → AI 生成研究报告 → 导出 Word/Excel 的完整流程。

## 1. 场景定义

**目标用户**：西南政法大学新闻学院老师 / 研究生

**研究问题**：分析 2025 年上半年（1-6 月）重庆 39 个区县在央级 + 省级媒体上的"乡村振兴"主题报道，输出包含数据简报 + 主题分布 + 区县分布 + 媒体层级分析的学术研报。

**成功标准**：
- 端到端 ≤30 分钟（不含 AI 生成等待）
- 输出 Word + Excel + 在线 HTML 三种形态
- 报告含数据驱动的结论（不是 LLM 主观写作）

## 2. 前置准备（一次性 / 已配置可跳过）

### 2.1 登录系统

访问 `http://localhost:3000`（开发环境）→ 用学校账号登录。

### 2.2 验证基础数据已就绪

打开侧栏 → 展开"研究"菜单 → 依次点开 3 个 admin 页面，**确认数据齐全**：

| 页面 | URL | 应看到 |
|---|---|---|
| 主题词库 | `/research/admin/topics` | **16 个**主题（含"乡村振兴"、"美丽中国"、"长江保护"等）|
| 媒体字典 | `/research/admin/media-outlets` | **113 个**媒体（央级 12 + 省市 8 + 区县 40 + 行业 12 + ...）|

### 2.3 检查"乡村振兴"主题的关键词

进 `/research/admin/topics` → 点击"乡村振兴" → 应看到关键词列表（如：乡村振兴 / 农业现代化 / 脱贫攻坚 / 美丽乡村 / 农村人居环境 ...）

如果只有 1-2 个关键词覆盖太窄 → 点"添加关键词"补 5-10 个相关词，命中率会提升。

## 3. 配置数据源（5-15 分钟）

> 数据源决定**从哪里抓内容**。一份研究报告通常需要 2-4 种源类型组合，覆盖央媒 / 省媒 / 社媒 / 关键词搜索全谱。

### 3.0 6 种源类型一览（实际 UI 显示的就是这 6 种）

进 `/data-collection/sources` → 点右上"新建采集源" → Step 1 看到 6 张卡片：

| 卡片 | type 值 | 适合场景 | 学术研报推荐度 |
|---|---|---|---|
| **聚合榜单 (TopHub)** | `tophub` | 抓微博/抖音/小红书/B 站等 10 平台**实时热榜** | ⚠️ 偏热点不偏学术，慎选 |
| **关键词搜索 (Tavily)** | `tavily` | 关键词全网搜，**支持时间窗 + 站点过滤** | ⭐⭐⭐ **首选** |
| **单 URL 深读 (Jina Reader)** | `jina_url` | 抓**指定 1 个 URL** 转 Markdown 全文 | 仅补充用 |
| **列表抓取 (正则或 CSS)** | `list_scraper` | 从新闻列表页**爬文章链接**（人民日报列表 → 文章 URL） | ⭐⭐⭐ 重要 |
| **RSS / Atom 订阅** | `rss` | 订阅 feed 地址（央媒大都有 RSS） | ⭐⭐⭐ 最稳，**首选** |
| **tikhub.io 社媒搜索** | `tikhub` | 抓微博 / 小红书 / 抖音 / 知乎 / 微信视频号 关键词搜索（半年内） | ⭐⭐ 看场景 |

### 3.1 推荐：学术研报最优组合

「**RSS + 列表抓取 + Tavily**」三件套通常足够覆盖央省级权威媒体：

```
RSS 订阅      ──→ 抓人民日报、新华社、央视、重庆日报等 RSS（最稳定）
列表抓取      ──→ 抓没 RSS 的网站（如某些政府网站）
关键词搜索    ──→ 用关键词追加扫全网漏网文章
```

如果论文要求覆盖**社媒舆情**（如分析微博传播），再加一个 tikhub 微博源。

### 3.2 配置 RSS 源（推荐起点 — 央媒 RSS）

**Step 1 选源类型**：点 **「RSS / Atom 订阅」** 卡片

**Step 2 配置参数**：
- **Feed URL**：填央媒 RSS 地址（**复制时不要带反引号 ` 或引号**），例如：
  ```
  http://www.people.com.cn/rss/politics.xml      ← 人民日报政治频道
  http://www.xinhuanet.com/politics/news_politics.xml  ← 新华网政治
  ```
  重庆日报需要找官方 RSS 入口（部分省市媒体未提供 RSS，用 §3.3 列表抓取代替）
- **深读正文 (Jina)**：建议**勾选** — RSS 通常只给摘要，勾选后会对每条链接再调一次 Jina Reader 抓全文（重要！否则正文匹配率低）

> 💡 **从 markdown 文档复制 URL 注意**：本手册的 ` `…` ` 反引号是 markdown inline code 标记，复制时不要选中反引号本身。如不慎带入，2026-05-08 后系统会自动剥离（rss/jina_url/list_scraper 三种 adapter 的 URL 字段都加了 `preprocess` 自动 trim）。

**Step 3 调度与分类**：
- **调度频率**：选 "每天一次"（央媒发文有节奏，无需高频）
- **归属模块**：✅ **必须勾选「研究 (news)」** — 这是**关键步骤**！只有勾这个，抓回内容才会进 research 模块的池子
- **媒体归属 - 绑定媒体**：选对应媒体（如填的是人民日报 RSS 就选「人民日报」）— 兜底用，每条仍会自动识别真实出处
- 默认分级 / 区域：留空（识别器自动填）

**Step 4 命名与确认**：起名"人民日报 RSS - 时政"→ 完成

> 重复以上步骤为每个央媒 RSS 各建一个源（推荐 3-5 个 RSS）。

### 3.3 配置列表抓取源（补 RSS 没覆盖的网站）

**Step 1**：点 **「列表抓取 (正则或 CSS 选择器)」** 卡片

**Step 2 配置参数**：

- **列表页 URL**：填新闻列表页（如 `http://www.cqrb.cn/news/`）
- **提取模式**：二选一
  - `regex`（正则）：填 **文章 URL 模式**，如 `https?://www\.cqrb\.cn/n/\d+\.html`
  - `css`（CSS 选择器）：填 5 个选择器
    - `items`：列表项选择器（如 `.news-list .item`）
    - `title`：标题（如 `.title`）
    - `link`：链接（如 `a[href]`）
    - `date`：日期（如 `.date`，可选）
    - `summary`：摘要（如 `.summary`，可选）
- **每次最大文章数**：默认 10（可调 1-100）
- **是否深读全文**：勾选则用 Jina Reader 抓正文（建议勾选）

**Step 3 + 4**：同 §3.2

> 💡 **CSS 模式 vs regex 模式**：现代网站推荐 css（更稳定）；老式静态 HTML 用 regex 更简单。

### 3.4 配置关键词搜索源（Tavily — 全网漏网捕捞）

**Step 1**：点 **「关键词搜索 (Tavily)」** 卡片

**Step 2**：
- **关键词**（多选）：填 5-10 个核心关键词，如：
  - `重庆 乡村振兴`
  - `重庆 农业现代化`
  - `重庆 美丽乡村`
  - `涪陵 乡村振兴`（针对性区县关键词）
- **时间窗**：选 `30d`（最近 30 天）OR `7d`
- **站点过滤**（可选）：限定央省级域名，如 `people.com.cn,xinhuanet.com,cqrb.cn`
- **每次结果上限**：默认 8（可调 1-20）

**Step 3 + 4**：同 §3.2

> 📌 Tavily 默认全网搜索；填 `includeDomains` 过滤可大幅提高研报数据可信度。

### 3.5 配置 Tikhub 社媒源（可选 — 抓微博/小红书/抖音/微信视频号/知乎）

**前提**：你需要在 [tikhub.io](https://tikhub.io) 注册并拿到 API key（按调用付费）。在 `.env.local` 配置 `TIKHUB_API_KEY=xxx`。

**Step 1**：点 **「tikhub.io 社媒搜索」** 卡片

**Step 2**：
- **平台**（单选）：从下拉选 微博 / 小红书 / 抖音 / 微信视频号 / 知乎 之一
- **关键词**（多选 1-20）：如 `重庆 乡村振兴`
- **时间窗**：默认半年内（tikhub 限制最长 6 个月）
- **每次最大页数**：默认 1（每页 ~20 条）

**Step 3 + 4**：同 §3.2

> 💰 **成本提醒**：tikhub 按 API 调用付费（详见 `src/lib/collection/adapters/tikhub/pricing.json` 的实测单价）。先用 1 页（~20 条）试跑，看 `/data-collection/sources/[id]` 的"已花费"数字，确认成本可控再调大。

### 3.6 配置聚合榜单 (TopHub) 源（实时热榜场景，研报谨慎）

**Step 1**：点 **「聚合榜单 (TopHub)」** 卡片

**Step 2**：
- **平台**（多选）：从 10 个里选 — weibo / zhihu / baidu / douyin / toutiao / 36kr / bilibili / xiaohongshu / thepaper / weixin

**Step 3 + 4**：同 §3.2，但**调度建议「手工触发」**（热榜实时性强，不需要定时跑）

> ⚠️ TopHub 内容偏**实时热点**（爆款 / 争议事件），与学术研报追求的"权威 + 长时段分布"诉求不完全匹配。如果研究主题恰好是热点（如某政策发布后的舆情），可加入；常规历史回顾性研报建议跳过。

### 3.7 单 URL 深读 (Jina Reader) — 补具体文章

**用途**：你已经有一篇关键文章 URL，要把它正文抓回研究池。**不适合定时采集**（每次只抓 1 个 URL）。

**Step 1**：点 **「单 URL 深读 (Jina Reader)」** 卡片  
**Step 2**：填该文章的 `网页 URL`

**Step 3 调度建议「手工触发」** — Jina Reader 适合"我要这篇"的一次性补采。

### 3.8 触发首次采集（验证源可用）

每个新建的源进 `/data-collection/sources/[id]` 详情页 → 点"立即运行" → 等 30s-2min → 看"已采集 X 条"数字增长。

失败时看详情页"运行日志"，常见错误：

| 错误 | 排查 |
|---|---|
| `Failed to fetch RSS` | RSS URL 失效 / 网站 robots 拒绝 → 换 URL |
| `Tavily quota exceeded` | Tavily API 配额耗尽 → 升级或换 key |
| `tikhub 401 unauthorized` | API key 错或过期 → 在 `.env.local` 更新 `TIKHUB_API_KEY` |
| `parse error: 0 articles found` | 列表抓取的 selector 不匹配实际页面 → 用 DevTools 看页面结构调整 |
| `partial failures: 5/10 succeeded` | 部分 item 抓失败（网络抖动）→ 通常忽略，下次跑会重试 |

> 💡 **绑定媒体不是限制**：每条采回的内容会**自动识别真实媒体出处**（URL host 匹配 / 公众号名 / 媒体字典 fuzzy match）。绑定媒体仅在识别失败时作兜底。所以一个 Tavily 关键词源就算"绑定媒体"留空也能正确归类各家媒体。

## 4. 等数据自动入库 + 自动打标（Inngest 后台跑）

每条 collected_item 入库后会触发 Inngest 函数 `research-annotate-collected-item`，**全自动**做：

1. **主题匹配**：扫文章标题 + 正文，匹配 16 主题词库的关键词 → 写 `research_collected_item_topics`
2. **区县匹配**：扫文章正文，匹配 39 个重庆区县名 → 写 `research_collected_item_districts`
3. **媒体识别**：根据 URL host 在 `media_outlet_dictionary` 找匹配 → 填 `collected_items.outlet_id`

> ⏱ **打标延迟**：通常 5-30 秒每条。批量入库时排队，1000 条约 5-15 分钟跑完。

**校验**：进 `/research` 高级模式 → 任意条件 → 看返回结果列表 → 看每条文章的"主题"、"区县"、"媒体"列是否都有值。如果大量为空 → 说明打标没跑（看 Inngest dashboard `/api/inngest/dev` 找错）。

## 5. 高级检索（核心使用）

### 5.1 进入

侧栏 → "研究" → "检索工作台"（即 `/research`）

### 5.2 切换到"高级"模式

页面顶部默认是"简单"模式。点"高级"切到 11 字段构造器。

### 5.3 配置检索条件（针对本场景）

针对 "2025 上半年重庆乡村振兴的省级及以上媒体报道"，配 4 个条件 + 1 个侧栏过滤：

**手动条件 1：主题 = 乡村振兴**
- 字段：`主题`
- 操作符：`等于`
- 值：从下拉选 "乡村振兴"
- 与下条逻辑：`AND`

**手动条件 2：发布时间 between 2025-01-01 ~ 2025-06-30**
- 字段：`发布时间`
- 操作符：`在范围内`
- 值：日期范围选择器选 2025-01-01 至 2025-06-30
- 与下条逻辑：`AND`

**手动条件 3：媒体分级 = 央级**（多选用 OR 链）
- 字段：`媒体分级`
- 操作符：`等于`
- 值：央级 (central)
- 与下条逻辑：`OR`

**手动条件 4：媒体分级 = 省级**
- 字段：`媒体分级`
- 操作符：`等于`
- 值：省级 (provincial_municipal)
- 与下条逻辑：`AND`

**侧栏过滤：区县 = 涪陵区 + 渝中区 + ...**（多选）
- 在右侧侧栏"区县"多选下拉里选要分析的区县（可全选 39 个，或针对性选 5-10 个）

> 💡 **手动条件 vs 侧栏的区别**：
> - **手动条件**支持精确字段 + AND/OR 复杂逻辑（最多 10 行）
> - **侧栏**是快速 chip / multi-select，组内 OR、跨组 AND（不限数量）
> - 两者**取交集**生效

### 5.4 点"开始检索"

下方表格显示命中结果。每条文章一行，可看：标题 / 媒体 / 时间 / 原文链接。

**关注顶部"命中 X 条"数字**：

| 命中数 | 操作 |
|---|---|
| **0 条** | 条件太严 → 放宽时间窗 / 增加主题关键词 / 减少区县过滤 |
| **1-100 条** | 适合快速验证 → 直接生成报告 |
| **100-500 条** | **报告最佳区间** ✓ |
| **>500 条** | 必须缩窄（"生成报告"按钮 disabled）→ 加更具体条件 |

### 5.5 让小研帮你写检索条件（可选）

不想手动配？打开右下角对话中心 → 选"小研" → 输入：

> "我想看 2025 上半年重庆乡村振兴的省级及以上媒体报道"

小研会输出建议的 conditions[] + sidebarFilter，并提供"一键填入 A4 高级检索"按钮 → 点击就自动跳转 + 预填条件。

## 6. 生成研究报告（入口 1：高级检索 → 报告）

### 6.1 命中 ≤500 条时的"生成报告"按钮

检索结果页顶部（命中数旁边）会出现"生成报告"按钮。

### 6.2 点击 → 弹 dialog

| 字段 | 输入 |
|---|---|
| **报告标题** | `2025 上半年重庆乡村振兴报道分析` |
| **主题描述**（可选） | `分析央省级媒体在乡村振兴主题下对重庆 39 个区县的报道分布与传播特征` |

### 6.3 确认 → 自动跳 `/research/reports/[id]`

报告进入**生成中**状态，5 个阶段顺序跑（共 30-90s）：

```
[阶段 1] 数据聚合         ~3s   读 hitItemIds → 计算主题/区县/媒体层级/时间趋势
[阶段 2] 数据简报         ~50ms 模板插值 plain text 数据简报
[阶段 3] 小研撰写中       ~30-50s LLM 调用：背景段 + 简报润色 + 研究发现
[阶段 4] 渲染 HTML        ~100ms 拼接最终 HTML
[阶段 5] 生成 Word        ~5-15s docx lib + 上传 Storage
[阶段 5] 生成 Excel        ~3-8s 同上（5 步并行）
```

页面会自动 polling，progress 文案实时更新。无需手动刷新。

> ⚠️ **AI 段落降级 banner**：如果 LLM 调用 3 次都失败（断网 / API key 错），自动用模板兜底生成默认段落，并在报告顶部显示 红黄色 banner "AI 段落降级，可重新生成"。点 banner 内"重新生成"按钮即可重新跑 LLM。

## 9. 报告版本管理（快照工作流）

### 9.1 默认是"覆盖"模式

每次点"重新生成"会**清空当前报告内容**重新跑。这是默认设计 — 学术写作中间版本通常不重要。

### 9.2 关键节点用快照锁版本

**何时建快照**：
- 导师 review 通过的版本
- 投稿前的最终定稿
- 给同事看的中间分享版

**操作**：
1. 报告页顶部 → "另存为快照"
2. dialog 输入快照名（如"导师 v1 反馈版"、"投稿前定稿"）
3. 确认 → 系统复制当前报告内容到新行（parentReportId 指回母版）
4. 顶部"快照 (N)" 按钮计数 +1，点击展开快照列表，可跳到任意快照详情

> 📌 **快照不能"重新生成"**（保护历史）。母版可以继续重新生成 / 改动，快照永远定格在那个时刻。

### 9.3 跨快照对比

V1 没做对比 UI。但快照都有独立 URL（`/research/reports/{snapshotId}`），可以左右两个浏览器标签同时打开对比措辞 / 数字。

## 11. 故障排查

### 11.1 检索结果是 0 条

| 可能原因 | 排查 |
|---|---|
| 没数据源在抓 | `/data-collection/sources` 看有几个源 enabled=true |
| 自动打标没跑 | 进任意命中文章详情页看 "主题"、"区县" 字段是否有值；都为空 → Inngest 没跑 |
| 时间窗太窄 | 放宽 `publishedAt` between |
| 主题匹配率低 | `/research/admin/topics` 给该主题加更多关键词 |
| 自动打标历史回填没跑 | 终端跑 `pnpm tsx scripts/research-backfill-annotate.ts` |

### 11.2 报告卡在"生成中"超过 5 分钟

| 排查 | 操作 |
|---|---|
| Inngest dev server 没启动 | 终端跑 `npx inngest-cli@latest dev -u http://localhost:3000/api/inngest` |
| LLM API key 错 | 看 `.env.local` 的 `OPENAI_API_KEY` 是否正确（项目用 DeepSeek API） |
| Inngest 内部错 | 打开 `http://localhost:8288` Inngest dashboard 看 `research/report-generate` function 的 run 状态 |
| 网络断 | LLM 重试 3 次都失败 → 自动降级到模板段落 + isAiFallback banner |

### 11.3 Word 打开报"未知错误"

| 排查 | 操作 |
|---|---|
| 中文字体缺失 | Windows + Office 默认有宋体/黑体/楷体；macOS 客户需确保有 |
| 文件被防病毒拦截 | 检查 Word 信任中心设置 |
| 文件签名 URL 已过期 (>24h) | 回报告页重点导出按钮重新签名 |

### 11.4 媒体识别全空（outlet_id 都为 NULL）

| 可能原因 | 排查 |
|---|---|
| `media_outlet_dictionary` 为空 | 跑 `pnpm tsx scripts/run-seed-media-outlets.ts` |
| 数据源的内容不在白名单内 | `/research/admin/media-outlets` 加新媒体 |
| 默认兜底没设 | 编辑数据源 → 配置"绑定媒体"作兜底 |

### 11.5 登录后跳转回登录页（cookie 丢失）

| 排查 | 操作 |
|---|---|
| `AUTH_SESSION_SECRET` 改了 | `.env.local` 确保不变；改了所有人都得重新登录 |
| 浏览器清过 cookies | 重登 |
| 跨设备使用 | 每设备独立登录，cookie 不互通 |

## 17. 设计哲学（理解系统的本质）

vibetide 的研究模块是**三层解耦设计**：

1. **采集层（Collection Hub）** — 不关心研究语义，只负责把内容稳定抓回来 + 写进 `collected_items` 池
2. **标注层（Annotation）** — Inngest 自动跑 topic / district / outlet 三个匹配，把内容**打标**让它能被检索
3. **应用层（Research / Report）** — 基于打标结果做检索 + 透视 + 报告

**为什么这样设计**：
- 采集层可独立扩展（加新平台、加新爬虫不影响下游）
- 标注规则可独立优化（改主题关键词不需要重抓）
- 报告 / 检索 / 后续分析都基于同一份打标结果，**单一真相源**

**所以遇到问题的排查顺序应该是**：
1. 数据有进 collected_items 吗？（看 /data-collection/items）
2. 打标跑了吗？（看任意一条 item 的 topics / districts 字段）
3. 检索能命中吗？（用宽松条件试一下 /research）
4. 报告能生成吗？（看 Inngest dashboard）

**任何一步断掉都会导致下游"看似没数据"。** 一旦理解这个分层，故障排查就有清晰的顺序。
